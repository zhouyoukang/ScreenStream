# ScreenStream_v2 标准流程（PROCESS）

> 目的：把你要求的“一次性闭环”变成可复用流程，减少中断与返工。

## 0) 核心闭环（固定顺序）

1. **证据定位**：找到权威入口（代码/配置/文档）
2. **差异/根因**：列出冲突点、重复点、不可维护点
3. **方案/ADR**：涉及端口/入口/鉴权/架构取舍必须 ADR
4. **实现**：最小变更，优先兼容迁移
5. **验收**：可复查、可复现（端口/URL/功能回归）
6. **归档**：更新 docs/STATUS + README 索引 + skills（如适用）

## 1) 何时必须写 ADR

- 端口/入口统一（例如 Input 与 MJPEG 的 HTTP Server）
- 对外 API 变更（路由/参数/协议）
- 合并策略（Quest/上游差异如何开关化）
- 安全策略（Pin/鉴权/CORS/反代）

ADR 目录：`docs/adr/`

## 2) 证据包（最小可复查）

每次推进一个问题，至少产出：

- 入口路径（文件路径 + 关键类/函数）
- 配置入口（Setting/默认值/URL 参数）
- 影响面（哪些模块会受影响）
- 验收步骤（最少 3 条：本地/局域网/目标设备）

## 3) 文档收敛规则

- 同主题只允许 1 个权威入口：
  - 研发管理：`docs/README.md`
  - 状态：`docs/STATUS.md`
  - 决策：`docs/adr/ADR-*.md`
- 临时笔记允许存在，但成熟内容必须回写到权威入口。

## 4) Skills 的使用方式（不平台化）

- Skills 是 **可复用工作流（Runbook/Checklist）**，不是运行时功能。
- 触发条件：高频重复、容易漏项、步骤可标准化。
- 目录：`skills/`（详见 `docs/SKILLS.md` 与 `skills/README.md`）。
