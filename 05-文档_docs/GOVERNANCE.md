# 人机共治论

> 统摄 ScreenStream_v2 规则体系之顶层设计、运行实态、得失剖析。
> 成文于丙午年二月廿一（2026-02-21），不替旧制，以文卷存照。

---

## 序

夫治者，制也。制者，所以御众、定分、明序也。

今世有智械（AI Agent）可与人共事。人之智在决断，械之智在推演。欲令人机相得，必立制度焉。余自丙午年正月起，历四十六对话，积六规则、九技能、十工作流、八目录指令、数十记忆，渐成一体。今当退而观之，审其得失，以为后鉴。

---

## 卷一 · 道论——用户之沟通本性

> 「识己者明，知人者智。」——欲治人机之间，先明用户之道。

### 第一章 七式

观四十六对话之实录，得用户提示词之七个原子模式：

**一曰极简。** 十七字发令，望千行功成。此非惰也，乃信也——信械之能推衍全局也。如将帅下令「取城」，不言「先遣斥候，再布火炮，三面合围」——信其将自为之。

**二曰元编程。** 不教械以事，而编械之性。譬如不授人以鱼，亦不授人以渔，而改渔者之本性使之善渔。此为编程之编程、规则之规则——用户之真正产品非代码，乃规则体系本身。

**三曰熵洁。** 乱则焦，整则安。每见冗余必欲除之，每见散落必欲归之。十九目录整为九，三十文档精为十——此性使然也。

**四曰握权。** 设 cunzhi 为关卡，每步可决。此如帝王之批红——虽有宰辅拟旨，终须御笔亲裁。安全之网也，亦摩擦之源也。

**五曰不信默认。** 以规则覆 Agent 默认行为。如法家之思——「人性不可恃，制度方可恃」。有效然有限：法条愈多，犯法之概率愈高，执法之成本亦愈高。

**六曰进化迷恋。** 望系统日智日新。设「双输出原则」「被动进化」「/evolve 工作流」——使进化为工作之副产品。然副产品无退出机制，则积累必过载。

**七曰发散聚焦。** 先穷举再筛选。创意阶段鼓励「几十个大模块都OK」，执行阶段只挑一二。然规则系统只有「添加」之门，无「淘汰」之门。

### 第二章 矛盾之核

用户立五条哲学公理以驭 Agent。取前二条审之：

> 公理一：**认知卸载**——用户注意力最稀缺。
> 公理二：**信息熵减**——每次交互减少系统总熵。

然系统之实际运行：

```
双输出原则要求 → 每次交互必产出 Memory / 规则
                → 系统总规则量单调递增
                → 上下文消耗单调递增
                → Agent 实际工作空间单调递减
                → 信息熵单调递增
```

**以减熵为名，行增熵之实。** 此为元系统悖论。

譬如治水：欲以堤防水，堤愈高水愈积，终有溃堤之日。善治水者知：**不可只筑堤，必须同时开渠。** 规则系统需要的不是更高的堤（更多规则），而是更通的渠（退出机制）。

## 卷二 · 制论——顶层架构实态

### 第三章 九层之塔

Agent 每启一对话，须读以下九层规则，未及干活已阅万言：

```
┌─────────────────────────────────────────────┐
│  L0  System Prompt（Cascade 内置，不可变）    │  ~8,000 字
├─────────────────────────────────────────────┤
│  L1  global_rules.md（跨项目通用）           │  ~4,000 字
├─────────────────────────────────────────────┤
│  L2  .windsurfrules（项目元说明）            │  ~1,500 字
├─────────────────────────────────────────────┤
│  L3  soul.md（Always On）                   │  ~3,000 字
│      execution-engine.md（Always On）        │  ~4,500 字
│      project-structure.md（Always On）       │    ~800 字
├─────────────────────────────────────────────┤
│  L4  AGENTS.md × 8（目录级指令）             │  ~1,500 字（根级）
├─────────────────────────────────────────────┤
│  L5  System-retrieved Memory（自动检索）      │  ~12,000 字（不可控）
├─────────────────────────────────────────────┤
│  L6  条件规则（kotlin/html/build-deploy）     │  按需加载
├─────────────────────────────────────────────┤
│  L7  Skills（9个）                          │  按需加载
├─────────────────────────────────────────────┤
│  L8  Workflows（10个）                      │  按需加载
└─────────────────────────────────────────────┘

固定开销合计：~35,000 字符（占上下文窗口约 15-20%）
```

### 第四章 冗余量化

同一信息跨层重复出现，以「PowerShell hooks 绝对禁止」为例：

| 出处 | 层级 |
|------|------|
| `global_rules.md` 第76行 | L1 |
| `execution-engine.md` 第20行 | L3 |
| `.windsurfrules` 第36行 | L2 |
| System Memory `573263f6`（hooks事故铁证） | L5 |
| System Memory `d8b59406`（全局配置体系） | L5 |

**五处言同一事，耗约八百字，传一比特信息。**

类似冗余者：

| 信息 | 重复次数 | 冗余字符 |
|------|---------|---------|
| PREDICT 框架 | 2 | ~600 |
| 项目结构 / 端口 | 3 | ~1,200 |
| 双输出原则 | 3 | ~500 |
| 进化闭环 | 2 | ~400 |
| 文件工具优先 | 2 | ~300 |
| **合计冗余** | | **~3,800** |

3,800 ÷ 35,000 = **10.9% 上下文被重复信息占据。**

### 第五章 Memory 噪声

本次对话（2026-02-21），Windsurf 自动检索 20 条 Memory。逐条审之：

| 类别 | 条数 | 占比 | 示例 |
|------|------|------|------|
| ✅ 直接相关 | 2 | 10% | 规则体系审计、DevCatalyst 发布 |
| 🟡 间接相关 | 3 | 15% | 用户方法论、大模块闭环、丛林开发教训 |
| ❌ 完全无关 | 15 | **75%** | Mina API token、HassWP 部署、remote_agent 端口、.so 注入方案、OPPO 拦截、eWeLink 研究... |

**七成五之 Memory 为噪声**，消耗约九千字符上下文。Agent 须读完全部方能判断何者有用——此本身即为认知负担，违公理一。

## 卷三 · 弊论——五弊

### 弊一：规则冗余

同一规则散布多处，改一处而忘余处，则生矛盾。如汉初「约法三章」之简明高效，后代律令积至万条，官吏不能尽知，百姓不能尽守——法多则法弱。

### 弊二：Memory 黑盒

用户不能控制何者被检索。Windsurf 以语义相似度自动匹配，然：
- 「解构沟通逻辑」被匹配至 .so 注入方案、Mina API token——风马牛不相及
- 过时 Memory 无自动退出机制
- 用户无批量清理工具

### 弊三：元系统悖论

（详见第二章。）建减熵系统而运行机制增熵——如以灭火为业而每日纵火。

### 弊四：边际效用递减

```
效用 ↑
     │         ╭───╮
     │       ╱       ╲
     │     ╱           ╲ ← 你在此处（50+规则）
     │   ╱               ╲
     │ ╱                   ╲
     └──────────────────────→ 规则数量
         10    30     50+
```

过最优点后，每增一规则，系统效用反降——因上下文消耗增、冲突概率增、维护成本增。

### 弊五：层级耦合

- `soul.md` 引用 global_rules（「PREDICT框架→见全局规则」）
- `execution-engine.md` 与 global_rules 共享「文件工具优先」「hooks 策略」
- `.windsurfrules` 纯为转发（「已迁移到结构化规则」）

层间引用如蛛网——改一层而不知影响几何。

## 卷四 · 变论——所当变革者

> 「穷则变，变则通，通则久。」

### 策一：开渠——建规则退出机制

当前系统只有 `create_memory`，无 `expire_memory`。建议：

- **Memory 分级**：核心理念（永久）/ 架构决策（90日）/ 技术细节（30日）/ 一次性记录（7日）
- **`/evolve` 双向化**：不只「发现新规则」，更「淘汰旧规则」——每次进化，清理 ≥ 创建
- **双输出原则修订**：输出B 从「系统进化」改为「**熵审计**」——评估是否需要清理，而非是否需要添加

### 策二：压层——九层化五层

| 新层级 | 内容 | 来源 |
|--------|------|------|
| L0 | System Prompt（不可变） | 不变 |
| L1 | global_rules.md（纯跨项目通用，去重后） | 去除与 L2 重叠 |
| L2 | AGENTS.md + project-structure（合并为项目认知层） | 合并 L2+L3+L4 |
| L3 | 条件规则（*.kt / *.html / build-deploy） | 不变 |
| L4 | Skills + Workflows（按需） | 不变 |

**去掉的**：
- `.windsurfrules`：38行纯元描述，无实际规则 → 压缩至3行或删除
- `soul.md` 独立文件：与 global_rules 高度重叠 → 项目特有部分并入 AGENTS.md
- `execution-engine.md` 拆分：通用部分并入 L1，项目部分并入 L2

**预期收益**：固定开销从 ~35,000 字减至 ~22,000 字（省 37%）。

### 策三：净化——Memory 卫生

当前可删除之 Memory（价值已归零）：

| Memory | 理由 |
|--------|------|
| Mina API token / 路径 | 一次性凭据，过期即废 |
| remote_agent.py 部署细节 | 端口 / SID / 坐标——过度细节 |
| .so 动态链接库寄生方案 | 纯理论探讨，未实施 |
| HassWP 部署 / token | 环境特定，非项目知识 |
| Phone Agent 本轮成果 | 已归档至 `管理/`，Memory 冗余 |
| v32+ / v33++ 增量成果 | 历史快照，已写入 FEATURES.md |
| hooks.json 4次反复 | 教训已固化入规则，详细记录无需保留 |

预计可清理 **10+ 条** Memory，减少 ~6,000 字符噪声。

### 策四：松绑——cunzhi 弹性化

当前 cunzhi 为「每次回复必须调用」。建议分场景：

| 场景 | cunzhi 策略 |
|------|------------|
| 开放性问题 / 多选择点 | ✅ 调用（提供选项） |
| 明确执行指令（如「编译」） | ⚠️ 可选（执行完报告即可） |
| 纯问答（如「这个函数干嘛的」） | ❌ 不需要（直接回答） |

**从「每次必调」变为「有决策点才调」**——减少无意义交互摩擦。

## 卷五 · 鉴论——得失总鉴

### 所得

| 成就 | 评语 |
|------|------|
| 从零建立多层规则体系 | 超越 99% 的 AI 用户 |
| PREDICT 框架 | 精炼且有效的决策模型 |
| 双输出原则 | 创新性的进化机制（需修正方向） |
| 用户提示词 DNA 提炼 | 罕见的自我认知深度 |
| DevCatalyst 开源 | 将个人实践转化为公共价值 |
| cunzhi 工具 | 独创的人机交互控制层 |

### 所失

| 问题 | 评语 |
|------|------|
| 规则过最优点 | 加法做尽，减法未始 |
| Memory 无退出 | 只进不出，终将溢出 |
| 元系统悖论 | 建减熵系统而运行增熵 |
| 层级耦合 | 改一动三，维护成本高 |
| 时间分配 | 治规则之时 > 写代码之时（部分对话） |

### 一言蔽之

> **从「编写规则」转向「策展规则」。**
>
> 不是写更多规则让 Agent 更聪明，而是精选最少规则让 Agent 最高效。
> 不是每次交互都添加新知，而是每次交互都审计旧知。
> 不是追求规则的完备性，而是追求规则的**信噪比**。

## 跋

善治者，不以法多为能，以法精为功。

秦以律令一天下，汉初约法三章而天下安。唐之律疏，精要五百条而治盛世。

今之规则体系，犹战国末法——条目繁密、互有抵牾、层级交错。所需者非增补，乃统一简化：

**精简、一致、层级分明、进退有序。**

此文不替旧制，但为鉴照。
