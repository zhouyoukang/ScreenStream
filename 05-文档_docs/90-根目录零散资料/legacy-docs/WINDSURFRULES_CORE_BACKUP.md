# ScreenStream_v2 核心规则文件备份

> 本文件为核心规则备份，不含MCP调用，用于新IDE环境的规则恢复。

## 【ScreenStream_v2 项目特定规则】

### 项目结构理解（强制按此路径定位）
```
ScreenStream_v2/
├── 01-应用与界面_app/           # 主应用入口
├── 02-通用组件_common/          # 共享模块
├── 03-流媒体_mjpeg/            # MJPEG流媒体
├── 04-反向控制_Input/          # 输入控制
├── 05-文档_docs/               # 项目文档（权威入口）
├── 06-技能_skills/             # 技能包目录
├── 07-实时传输_rtsp/           # RTSP协议
├── 08-网络通信_webrtc/         # WebRTC实现
├── 09-构建与部署_Build/        # 构建配置
└── api-services/               # API服务化架构
```

### API服务端口分配（固定，禁止冲突）
- Gateway: 8080 (统一入口)
- MJPEG: 8081 (流媒体服务)
- RTSP: 8082 (实时传输)
- WebRTC: 8083 (网络通信)
- Input: 8084 (反向控制)

### 权威入口优先级（必须按序查找）
1. **项目总入口**: `docs/README.md`
2. **模块索引**: `docs/MODULES.md`
3. **功能映射**: `docs/FEATURES.md`
4. **流程规范**: `docs/PROCESS.md`
5. **技能指南**: `docs/SKILL_HOWTO_CN.md`

## 【技能化工作流集成】

### 技能包应用规则
- **触发条件**: 遇到重复性、复杂性任务时必须先检查是否有对应技能包
- **技能位置**: `06-技能_skills/<skill-name>/SKILL.md`
- **执行顺序**: 证据定位 → 差异/根因 → 方案/ADR → 实现 → 验收 → 归档

### 常用技能包映射
- **API服务测试**: `skill-api-service-test/`
- **终端命令组织**: `skill-ssv2-terminal-runbook/`
- **输入链路统一**: `skill-ssv2-input-unify/`
- **合并计划生成**: `skill-ssv2-merge-plan/`
- **发布检查清单**: `skill-ssv2-release-checklist/`

### 技能执行约束
- 技能包只是工作流指导，不替代技术判断
- 涉及端口/入口/鉴权的必须先查阅或创建ADR
- 所有技能执行结果必须产生可复查证据包

## 【API服务化开发模式】

### 开发测试优先级
1. **后端API优先**: 优先通过RESTful API验证功能
2. **无APK测试**: 禁止以"需要APK测试"为由中断开发
3. **服务化思维**: 功能拆分为独立API服务考虑

### API开发标准流程
```bash
# 1. 服务健康检查
curl http://localhost:8080/health

# 2. 功能API测试
curl -X POST http://localhost:8080/<service>/<action> -d '{}'

# 3. 状态验证
curl http://localhost:8080/status

# 4. 集成测试
curl -X POST http://localhost:8080/start-all
```

### 常见问题快速解决
- **端口占用**: `netstat -ano | findstr ":808x"` → `taskkill /PID <pid> /F`
- **服务启动失败**: 检查 `api-services/logs/` → Java版本 → JAR文件完整性
- **API调用失败**: 健康检查 → 网络连通 → 服务日志

## 【模块化管理标准】

### 模块定位标准流程
1. **先查文档**: `docs/README.md` → `docs/MODULES.md`
2. **确定模块**: 根据功能映射到中文目录
3. **找到入口**: 代码入口 + 配置入口 + 文档入口
4. **验证范围**: 确认影响的其他模块

### 模块间依赖规则
- **向上依赖**: 应用层可依赖通用组件
- **平级隔离**: 流媒体模块间应保持独立
- **接口优先**: 模块间通信优先使用明确接口

### 架构决策要求
- **端口/入口/鉴权**: 必须先写ADR再实现
- **跨模块修改**: 必须评估影响面并记录
- **构建/签名/发布**: 禁止隐式修改关键配置

## 【执行模式控制】

### 一次性闭环执行（强制）
- 默认按"最优路线"一次性推进到可验收产物
- 禁止执行过程中让用户选择方向或分段确认
- 禁止询问：按哪个方向走/是否允许/是否手动操作
- 工具调用次数控制：同一文件/主题尽量一次性完成

### 证据收集优先模式
- 写入前先用只读工具收集所需信息
- 多个独立只读操作可并行
- 写操作必须串行且合并
- 同一文件修改必须合并为一次完整写入

### 终端执行防卡顿硬规则（必须遵守）
- 禁止“长脚本/多段逻辑/循环/大段管道输出”一次性塞进终端；必须拆成 3-6 条短命令。
- 禁止并行命令组（并行会放大卡顿与输出竞争）；同一阶段只执行一组短命令。
- 默认使用只读证据命令；任何会下载/构建/安装的命令必须单独成组。
- 命令输出必须可控：优先用 `findstr`/`Select-Object -First N` 截断；禁止直接 `dumpsys/logcat` 全量输出。
- HTTP 探测在 Windows 必须用 `curl.exe`（避免 PowerShell `curl` 别名造成不可控输出）。
- 发现卡顿/超时：立刻切换到“更短命令 + 更少输出”的方案，禁止在原方案上加码。

### 高风险动作处理
- 高风险包含：联网/安装依赖/写磁盘/杀进程/改端口/改签名
- 先完成证据定位/方案/实现草案/验收说明
- 在结尾一次性列出完整命令组等待确认

## 【验收和归档标准】

### 必达验收标准
- **功能验收**: 通过API测试验证功能正常
- **文档更新**: 更新 `docs/STATUS.md` 记录进展
- **证据保留**: 关键输出、错误日志、验证结果
- **规则遵循**: 按技能包流程执行并记录

### 归档要求
- **STATUS更新**: 记录当前状态和下一步行动
- **ADR记录**: 重要架构决策必须有ADR
- **技能完善**: 新的可复用流程写入技能包
- **问题记录**: 遇到的问题和解决方案记录

## 【使用方法】

### 新IDE环境规则恢复
1. 复制本文档内容到新IDE的 `.windsurfrules` 文件
2. 根据实际MCP工具调整结尾调用规则
3. 验证规则生效后开始开发
