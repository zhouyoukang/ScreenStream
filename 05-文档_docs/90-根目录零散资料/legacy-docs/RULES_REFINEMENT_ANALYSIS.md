# 规则文件精细化需求分析

## 📊 现状对比分析

### AIOT根规则 vs ScreenStream_v2规则

**AIOT/.windsurfrules (136行)**：
- 通用性执行规则
- 命令执行模式控制
- 错误模式防范
- 适用于所有子项目

**ScreenStream_v2/.windsurfrules (50行)**：
- 专用MCP工具 (`ymmokyyq_dev`)
- 固定panelId
- 简化版执行约束
- 缺少项目特定指导

## 🎯 精细化必要性评估

### ✅ 建议精细化的原因

#### 1. 项目特殊性
```markdown
ScreenStream_v2特有需求：
- API服务化架构：5个独立服务管理
- 模块化管理：中文目录结构
- 技能化工作流：可复用开发模式
- 无APK测试：后端直接验证流程
```

#### 2. 开发模式固化
```markdown
需要固化的开发模式：
- 模块定位：先查docs/README → MODULES → 具体入口
- 证据收集：先只读探查 → 一次性执行计划
- Skill应用：按既定工作流执行
- 验收标准：API测试 + 功能验证 + 文档更新
```

#### 3. 效率提升预期
- **减少重复分析**：项目结构、端口分配、模块职责固化
- **标准化流程**：API测试、服务管理、问题排查有统一套路  
- **避免常见错误**：端口冲突、权限问题、构建失败的预防机制

### ⚠️ 精细化风险评估

#### 潜在负面影响
1. **过度约束**：规则太细可能限制AI灵活性
2. **维护成本**：规则更新需要同步项目变化
3. **学习曲线**：新规则需要适应期

#### 缓解方案
1. **分层设计**：核心规则 + 可选指引
2. **版本管理**：规则变更有记录和回滚机制
3. **灵活性保留**：关键决策点仍保持AI自主判断
