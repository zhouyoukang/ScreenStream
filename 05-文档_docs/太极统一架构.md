# 太极统一架构 — 万物归一

> **道生一，一生二，二生三，三生万物。**
> 50个项目、15个服务、10个域 → 一个太极。
> 每个模块相互隔离（阴），又相互联系（阳），阴阳合一即太极。

---

## 一、五感解构 — 用户操作各模块的底层需求

> 剥离表面功能，追问：用户为什么要用这个模块？底层欲望是什么？

### 1.1 底层需求原子表

| # | 底层需求（原子） | 表面模块 | 用户真实欲望 |
|---|-----------------|---------|------------|
| N1 | **远程看到** | 公网投屏 / 亲情远程 / 电脑投屏 / SpaceDesk | 不在设备前也能看到它的画面 |
| N2 | **远程控制** | ScreenStream Input / phone_lib / remote_agent / hub.py | 不在设备前也能操作它 |
| N3 | **环境感知** | 智能家居 / 认知代理 / 机器狗传感器 | 知道周围环境的状态 |
| N4 | **环境改变** | 智能家居控制 / 机器狗动作 / 微信命令 | 远程改变物理世界 |
| N5 | **网络连通** | Clash / FRP / 阿里云 / 双电脑互联 | 任何设备任何时候都能连上 |
| N6 | **数据流转** | 二手书系统 / 三鲜同步 / 手机订单采集 | 信息自动流动，不需要人搬运 |
| N7 | **自动执行** | 认知代理工作流 / N8N / HA自动化 | 重复操作由机器代替 |
| N8 | **安全守护** | 双电脑互联/三界隔离 / Guardian | 设备不被误操作/入侵/丢失 |
| N9 | **知识创造** | ARGs论文 / 提示词升维 / 3D建模 | 用AI加速知识生产 |
| N10 | **商业运营** | 二手书+三鲜+微信公众号 | 用系统赚钱/管理业务 |

### 1.2 五感映射

```
视（看到）: N1远程看到 + N3环境感知 + N6数据流转的可视化
听（状态）: N5网络连通 + N8安全守护的告警
触（操作）: N2远程控制 + N4环境改变 + N7自动执行
嗅（预判）: N8安全守护 + N5网络连通的故障预判
味（评估）: N10商业运营的效果 + N9知识创造的质量
```

## 二、阴阳图谱 — 模块间隐藏的联系

> 表面：50个独立项目。实质：同一底层需求的不同实现。

### 2.1 阳（已建立的联系）

```
ScreenStream API ──→ phone_lib ──→ 智能家居面板
                                 ──→ 亲情远程控制
阿里云 Nginx ──→ FRP隧道 ──→ {远程中枢, 二手书, 投屏, 智能家居}
hub.py ──→ 14个子系统探测（但只读，不调度）
微信公众号 ──→ 智能家居网关 ──→ MiCloud/eWeLink ──→ 物理设备
```

### 2.2 阴（断裂的联系 — 本应连通却未连通）

| # | 断裂点 | 应有联系 | 影响 |
|---|--------|---------|------|
| B1 | **认知代理 ↛ hub.py** | 感知到用户意图后应自动路由到正确子系统 | 认知代理有感知无行动 |
| B2 | **hub.py ↛ 智能家居** | 统一中枢应能直接控制家居设备 | 控制入口分散 |
| B3 | **二手书 ↛ 微信公众号** | 订单状态变更应推送微信通知 | 用户需手动查看 |
| B4 | **5种投屏 互不感知** | 应有统一流管理器，按场景自动选择最优协议 | 用户需记住用哪个 |
| B5 | **机器狗 ↛ 智能家居** | Go1应作为移动传感器融入家居感知网络 | 机器狗是孤岛 |
| B6 | **AGI仪表盘 ↛ hub.py** | 两个"统一面板"各管各的 | 重复建设 |
| B7 | **3D建模 ↛ 3D打印** | 建模结果应直接推送到打印队列 | 手动导出+导入 |
| B8 | **VR系统 ↛ 智能家居** | MR Passthrough应能在VR中控制真实家居 | quest3 smart-home demo是静态mock |
| B9 | **所有服务 ↛ 统一告警** | 任何服务down应该推送通知（微信/TTS） | 只有watchdog检测阿里云 |
| B10 | **远程桌面 ↛ 认知代理** | 远程控制应利用认知代理的UIA感知做智能操作 | 纯键鼠模拟，不理解界面 |

## 三、未表述的底层需求（用户没说但一定需要）

> 十二因缘：从已表述的需求向上追溯，发现根因层的隐藏需求。

| # | 隐藏需求 | 推理链 | 证据 |
|---|---------|--------|------|
| H1 | **统一通知中心** | 15个服务各自沉默→用户必须轮询检查→认知负担巨大 | 只有watchdog有cron告警，其余全靠人工查看 |
| H2 | **事件驱动自动化** | 用户操作全是"拉"(pull)→应该是"推"(push)：事件来了→自动响应 | 三鲜订单来了无通知、设备离线无警告、构建完成无推送 |
| H3 | **跨设备工作流** | 单设备操作已自动化，但跨设备流程全靠人记忆 | 二手书：手机扫码→电脑入库→打印标签→上架，每步都要人切换 |
| H4 | **统一身份/会话** | 15个服务各自独立认证（或无认证）→安全风险+操作碎片 | 二手书有login、远程中枢有auth、其余全部裸露 |
| H5 | **数据洞察层** | 数据只流入不分析→大量信息沉睡 | 二手书有190路由但无BI面板、三鲜有1490取消订单但无分析看板 |
| H6 | **优雅降级链** | 主路径断了应自动切备路径 | 亲情远程有P2P→Relay降级，但其他模块没有 |
| H7 | **一键启动/一键关停** | 15个服务手动启停→每次开机浪费10分钟 | 各有start.bat但无统一编排 |
| H8 | **移动端统一入口** | 手机上只能通过浏览器访问散落的URL | 无PWA/APP聚合所有服务 |

## 五、统一方案 — 太极架构

### 5.1 核心设计原则

```
☰ 乾: 一个入口控制万物 → 升级hub.py为真正的太极中枢
☷ 坤: 每个模块保持独立 → 通过HTTP API松耦合，不改内部实现
☵ 坎: 断了就换路 → 每条通道有降级方案
☲ 离: 看到就理解 → 统一仪表盘取代双中枢
☳ 震: 数据自己流 → 事件总线让模块自动联动
☴ 巽: 渐进不激进 → 分4步实施，每步独立可验证
☶ 艮: 稳定不动摇 → 不改任何已稳定模块的内部代码
☱ 兑: 产出可复用 → 每步交付独立文件，不依赖会话
```

### 5.2 四步实施计划

#### Step 1: 统一健康探测 + 事件总线（最小可用）
**改动**: 升级 `hub.py` → 加入事件总线 + 统一告警
```
hub.py v3.0:
  + EventBus(内存级发布/订阅)
  + /api/events/publish  POST {source, type, data}
  + /api/events/subscribe GET (SSE长连接)
  + /api/alert POST → 微信公众号推送 / TTS播报
  + 定时健康检测(30s) → 状态变化时发事件 → 触发告警
```
**收益**: 任何服务宕机→微信通知+音箱播报。模块间可以发事件了。

#### Step 2: 认知代理接入（感知→决策→执行闭环）
**改动**: 认知代理 `/analyze` 输出 → hub.py EventBus → 自动路由到子系统
```
认知代理:
  + /webhook POST {event_type, action, target}
  + 意图识别后自动发布到EventBus
hub.py:
  + /api/dispatch POST {target_system, action, params}
  + 根据target路由到对应子系统API
```
**收益**: "用户打开了淘宝" → 认知代理感知 → 自动启动订单采集。

#### Step 3: 投屏统一 + 移动端入口
**改动**: 创建 `/stream` 统一入口页
```
stream.html:
  - 自动探测可用流源(SS MJPEG/WebRTC/电脑JPEG/亲情远程)
  - 按场景推荐最优协议
  - 一个URL访问所有画面
mobile.html:
  - 聚合所有服务的移动端入口
  - PWA离线可用
  - 底部Tab: 设备/投屏/家居/商城/系统
```
**收益**: 一个URL看所有屏幕。手机上一个APP管理一切。

#### Step 4: 跨设备工作流编排
**改动**: hub.py + 认知代理工作流引擎
```
预定义工作流:
  1. 二手书入库流: 手机扫码→电脑入库→打标签→更新库存
  2. 三鲜订单流: 新订单→微信通知→自动备货提醒
  3. 设备巡检流: 定时检测所有设备→异常告警→自动重启
  4. 开机启动流: 一键启动所有服务(按依赖顺序)
```
**收益**: 跨设备操作从手动记忆变成一键触发。

### 5.3 端口冲突解决

| 问题 | 方案 |
|------|------|
| 9800冲突(亲情远程 vs 公网投屏) | 公网投屏中继迁移到9810 |

### 5.4 双中枢合并

| 当前 | 合并后 |
|------|--------|
| hub.py :9000 (14系统探测+本地操作) | **太极中枢 :9000** (统一入口) |
| AGI仪表盘 :9090 (Skills/Rules/Health) | 合并为太极中枢的`/admin`标签页 |
