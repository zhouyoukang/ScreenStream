# ScreenStream → 超级个人 AI 助理 演进路线

> 核心理念：最小可用 → 逐步演进 → 最终形态

## 一、终极目标

**超级个人 AI 助理**：一个能完全理解用户意图、自主操控手机完成任何任务的系统。

用户只需说一句话（甚至不需要说），AI 就能：
- 看到手机屏幕（投屏）
- 理解当前界面（View 树 + OCR）
- 执行操作（触控 + 系统API）
- 自动化复杂流程（宏/规则引擎）
- 学习和适应（用户习惯建模）

## 二、为什么直接用代码操控手机行不通

用户经验总结（已验证的结论）：

| 方案 | 问题 |
|------|------|
| Python/ADB 脚本直控 | 效率低、逻辑复杂、每个场景要单独写代码 |
| AI 直接生成操控代码 | AI 不了解当前屏幕状态，盲操作失败率高 |
| Root + 代码注入 | 复杂、不通用、维护成本极高 |
| Xposed/Hook 框架 | 需要 Root/沙箱、兼容性差、法律风险 |

**有效的方案**：MacroDroid / Tasker 模式
- 把复杂操作"固化"为可复用的规则/宏
- 用户通过简单配置而非写代码来创建自动化
- 插件体系扩展能力

## 三、MacroDroid/Tasker 架构分析

### 核心架构（可借鉴）

```
┌─────────────────────────────────────┐
│  用户界面（可视化规则编辑器）          │
├─────────────────────────────────────┤
│  规则引擎：触发器 → 条件 → 动作       │
│  Trigger → Condition → Action       │
├─────────────────────────────────────┤
│  插件/技能系统（Plugin / Skill）      │
│  每个插件 = 一组触发器 + 动作         │
├─────────────────────────────────────┤
│  系统能力层                          │
│  AccessibilityService / 通知监听     │
│  / 传感器 / 位置 / 网络 / 日历...    │
└─────────────────────────────────────┘
```

### 映射到我们的系统

| MacroDroid 概念 | 我们已有的 | 需要建设的 |
|----------------|----------|----------|
| **动作 (Action)** | 42 个 API 端点 ✅ | 更多系统级操作 |
| **触发器 (Trigger)** | ❌ 暂无 | 事件监听系统 |
| **条件 (Condition)** | `/viewtree` `/windowinfo` ✅ | 条件判断引擎 |
| **宏 (Macro)** | ❌ 暂无 | 规则编排系统 |
| **插件 (Plugin)** | ❌ 暂无 | 插件加载框架 |
| **变量系统** | ❌ 暂无 | 全局/局部变量存储 |
| **UI 编辑器** | Web 前端 ✅ | 可视化规则编辑器 |

## 四、五阶段演进路线

### Phase 0: 投屏 + 基础控制（✅ 已完成 v30）
```
能力：看到手机屏幕 + 远程触控/键盘/导航
价值：远程协助家人
```

### Phase 1: 远程协助 + AI Brain（✅ 已完成 v31-v32）
```
新增：系统动作 + 增强手势 + View树分析 + 语义化操作
     + WebSocket实时触控 + 手机端查看器
价值：AI 可以"看到"并"理解"手机界面
现有：42 个 API 端点
```

### Phase 2: 自动化规则引擎（下一步）
```
新增：
  - 宏系统：将多个 API 调用编排成一个"宏"
  - 触发器：时间/应用切换/通知/电池/位置等事件
  - 条件判断：屏幕内容匹配/应用包名/系统状态
  - 变量系统：存储和传递中间数据
  - Web 端宏编辑器
价值：用户可以创建自动化规则，不需要写代码
示例：
  "收到微信消息 → 检查是否包含'到了吗' → 自动回复'马上到'"
  "打开支付宝 → 点击扫一扫 → 等待扫码"
```

### Phase 3: AI 决策引擎（中期）
```
新增：
  - 自然语言理解：用户说"帮我订个外卖" → 分解为操作步骤
  - 屏幕理解：OCR + View树 → 理解当前页面语义
  - 智能决策：根据屏幕状态决定下一步操作
  - 错误恢复：操作失败时自动重试或切换方案
  - 操作录制：用户操作一遍，AI 学会自动重复
价值：AI 能自主完成复杂多步任务
```

### Phase 4: 超级个人助理（长期）
```
新增：
  - 多设备协同：同时控制多台手机/平板
  - 用户画像：学习用户习惯和偏好
  - 预测性操作：在用户请求之前就准备好
  - 插件生态：第三方开发者可以贡献技能
  - 云端协同：跨设备同步规则和数据
价值：真正的个人 AI 助理
```

## 五、Phase 2 技术设计（立即可做）

### 5.1 宏系统数据结构

```json
{
  "name": "自动回复微信",
  "trigger": {
    "type": "notification",
    "app": "com.tencent.mm",
    "textContains": "到了吗"
  },
  "conditions": [
    { "type": "screenOn", "value": true }
  ],
  "actions": [
    { "type": "api", "endpoint": "/findclick", "params": { "text": "到了吗" } },
    { "type": "wait", "ms": 500 },
    { "type": "api", "endpoint": "/settext", "params": { "search": "", "value": "马上到" } },
    { "type": "api", "endpoint": "/findclick", "params": { "text": "发送" } }
  ],
  "enabled": true
}
```

### 5.2 触发器类型设计

| 触发器 | 实现方式 | 难度 |
|--------|---------|------|
| 应用切换 | `onAccessibilityEvent(TYPE_WINDOW_STATE_CHANGED)` | 🟢 低 |
| 通知到达 | `NotificationListenerService` | 🟢 低 |
| 定时触发 | `AlarmManager` | 🟢 低 |
| 屏幕开/关 | `BroadcastReceiver(ACTION_SCREEN_ON/OFF)` | 🟢 低 |
| 电池状态 | `BroadcastReceiver(ACTION_BATTERY_CHANGED)` | 🟢 低 |
| WiFi 连接 | `BroadcastReceiver(CONNECTIVITY_ACTION)` | 🟢 低 |
| 地理围栏 | `GeofencingClient` | 🟡 中 |
| 指定文本出现 | View树轮询 + 文本匹配 | 🟡 中 |
| 传感器事件 | `SensorManager` | 🟡 中 |

## 七、关键认知

1. **不要从零造轮子** — MacroDroid/Tasker 已经验证了自动化规则引擎的可行性，学习他们的设计模式
2. **我们的独特优势是 Web 端** — 用户通过浏览器就能创建和管理自动化规则，不需要在手机上复杂操作
3. **API 就是"动作库"** — 我们已有的 42 个端点就是 MacroDroid 中的"动作"，这是最大的存量资产
4. **AccessibilityService 是核心** — 它同时提供了：触发器（事件监听）、条件（View树查询）、动作（操作执行）
5. **最小可用逐步演进** — 先做宏系统（Phase 2），再做 AI 决策（Phase 3），最后做完整助理（Phase 4）
