---

## 1) 为什么“全量中文重命名”是高风险

全量重命名会带来：

- **构建链路风险**：Gradle module name、AndroidManifest、资源引用、反射/序列化、Proguard/R8、第三方配置等都可能隐式依赖包名/类名/路径。
- **合并成本暴涨**：后续你要持续吸收 Quest/上游差异，重命名会让 diff 失真、冲突增加。
- **历史追溯变差**：`git blame`、问题定位成本上升。

所以：除非你明确“愿意付出一次性迁移成本，并接受一段时间合并变慢”，否则不建议走“全量中文化重命名”。

---

## 2) 推荐路线（从低风险到高风险，逐层推进）

### L0（低风险，推荐默认）：文档层中文化（不动代码）

目标：不改目录/类名，也能让你“用中文管理与开发”。

做法：

- 以 `docs/PROJECT_MAP_CN.md` 为“中文项目地图”，维护：
  - 模块/功能 -> 入口文件 -> 配置 -> ADR -> Skill
- 在 `docs/MODULES.md`、`docs/FEATURES.md` 补齐“中文描述 + 入口定位点”。
- 每次你要做功能开发，我按 `docs/SKILL_HOWTO_CN.md` 的模板推进，产物收敛回 docs。

验收：

- 你能**不读英文**也能：
  - 定位入口（文件/类）
  - 知道改哪个 setting
  - 按 skill 跑完闭环（证据/实现/验收/归档）

### L1（中风险）：UI 文案中文化（可控，仍不动包名/类名）

目标：用户可见部分中文化，但不破坏工程结构。

做法（按模块逐个）：

- 只改 **可见字符串**（strings resources / Compose 文案）
- 保持 API/协议/路由不变

验收：

- UI 文案中文化
- 功能回归通过

### L2（高风险）：代码符号/目录重命名（不建议默认做）

只在你明确需要时执行（例如：要对外交付、团队协作强依赖中文命名）。

强制策略：

- **一次只改一个模块**（例如仅 `input/` 或仅 `mjpeg/`）
- 每次改动必须包含：
  - 迁移清单（改了哪些路径/包名/类名）
  - 可回滚策略（一个 commit 完整覆盖）
  - 验收命令组（构建 + 最小功能回归）
- 任何跨模块重命名都必须先 ADR

验收：

- `gradlew assembleDebug` 能过
- 关键功能（流/输入/UI）可用

## 3) 如果你坚持“整个项目文件夹重构”，正确做法是什么

把“重构”拆成两类：

- **管理重构（推荐）**：重构 docs/skills 体系、索引、流程、清单。
  - 你现在要的“最高效率”主要来自这里。
- **工程重构（高风险）**：移动/重命名源码目录、包名、模块名。
  - 这会显著拖慢 Quest/上游合并与日常开发。

建议：

1. 先完成“管理重构”的全覆盖（中文地图 + 功能索引 + skills 模板化）。
2. 等你后续要做的功能开发推进顺了，再挑选 1 个模块试点 L2。
