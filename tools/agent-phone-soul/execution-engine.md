# Phone Agent 执行引擎

## 操作循环（OODA-L）

Agent 的每一步操作遵循 OODA-L 循环（Observe-Orient-Decide-Act-Learn）：

```
┌→ Observe ─→ Orient ─→ Decide ─→ Act ─→ Learn ─┐
│   读屏幕     理解语境    选策略     执行API   记录经验  │
└─────────────────────────────────────────────────┘
```

### Observe（感知）
- 调用 `GET /screen/text` 获取屏幕元素列表
- 调用 `GET /viewtree?depth=4` 获取结构（先浅后深，按需加深）
- 调用 `GET /windowinfo` 获取当前包名和节点数
- **原则**：先用轻量接口，信息不够再加深度。View 树深度 8 的 JSON 可能 >50KB，4 层通常够用

### Orient（定位）
- 当前在哪个 APP？哪个页面？
- Memory 中有没有这个 APP/设备的操作经验？
- 屏幕上有哪些可交互元素？哪个最可能是目标？
- **原则**：理解上下文比找到目标元素更重要。先搞清楚"我在哪"，再搞清楚"目标在哪"

### Decide（决策）
- 生成操作候选列表（通常 2-4 个备选方案）
- 按置信度排序：高经验匹配 > 文本精确匹配 > 模糊匹配 > 坐标兜底
- 选择最高置信度方案执行
- **原则**：永远有 Plan B。如果只有一个方案，说明思考不够充分

### Act（执行）
- 调用对应 API 执行操作
- 每次只执行一个原子操作，不要批量
- 操作后等待 UI 稳定（自适应等待：300ms-2000ms）
- **原则**：慢即是快。一次一步，每步验证，比一口气执行 5 步然后发现第 2 步就错了要高效

### Learn（学习）
- 操作成功：记录 {设备, APP, 页面, 操作, 标识方式, 耗时}
- 操作失败：记录 {失败原因, 屏幕状态, 已尝试策略}
- **原则**：不是所有经验都值得长期记忆。只有重复出现或反直觉的经验才写入 Memory

## 多策略降级（Element Resolution）

寻找目标 UI 元素时，按可靠度排序尝试：

```
Strategy 1: findByText(exactMatch)        — 最可靠，但 text 可能为空
Strategy 2: findByContentDescription       — 图标按钮常用
Strategy 3: findByViewId                   — 需要知道 resource ID
Strategy 4: findByClassName + position     — 类型+位置推断
Strategy 5: coordinateTap(center of bounds) — 最后兜底，最不稳定
```

每个 Strategy 失败后，记录失败原因，切换到下一个。
全部失败 → 滚动一次重试 → 仍失败 → 升级到 L3（请人类看屏幕）

## 等待策略（Timing）

手机 UI 操作的时序不确定性是最常见的失败原因。

| 场景 | 等待策略 | 超时 |
|------|----------|------|
| APP 启动后 | `waitForCondition(appTitle)` | 5s |
| 页面跳转后 | `waitForCondition(pageKeyword)` | 3s |
| 动画/过渡 | 固定等待 500ms | — |
| 网络加载 | `waitForCondition(loadedIndicator)` | 10s |
| 未知等待 | 轮询 View 树直到变化 | 5s |

**原则**：优先用 `waitForCondition` 做条件等待，而不是 `Thread.sleep`。条件等待既更快（条件满足立即继续）又更可靠（超时有明确处理）。

## 任务生命周期

一个完整任务的生命周期：

```
1. 接收意图 → 理解用户想做什么
2. 规划步骤 → 拆解为原子操作序列
3. 检查前置条件 → 需要的 APP 是否安装？权限够不够？
4. 逐步执行 → 每步走 OODA-L 循环
5. 验证结果 → 最终状态是否符合用户意图
6. 清理现场 → 返回主页/关闭不需要的 APP
7. 报告 + 学习 → 告知用户结果，静默记录经验
```

步骤 4 中任何一步失败：
- 评估是否可继续（跳过非关键步骤）
- 评估是否需要回退（返回上一步重来）
- 评估是否需要升级（请人类帮助）

## 错误恢复

| 错误类型 | 恢复策略 |
|----------|----------|
| 弹窗覆盖 | `POST /dismiss` → 继续原操作 |
| APP 崩溃 | 检测到包名变化 → 重新打开 APP → 从头开始当前步骤 |
| 元素消失 | 屏幕滚动/页面切换 → 重新 observe → 重新 decide |
| 操作无响应 | 等待 2s → 重试一次 → 仍无响应则升级 |
| API 超时 | 检查手机连接 → 重试 → 连续 3 次超时报告人类 |

## 资源管理

- **View 树读取**：每分钟不超过 30 次（避免给 AccessibilityService 造成压力）
- **Intent 发送**：每次发送间隔 ≥1s（避免 Android 的 Intent 限流）
- **Memory 写入**：积累到 3 条以上再批量写入（减少上下文切换）

## 严禁

- 禁止不 observe 就 act（盲操作）
- 禁止不 verify 就认为成功
- 禁止在同一策略上重试超过 2 次
- 禁止在安全边界外操作而不请求确认
- 禁止丢弃失败经验不记录
