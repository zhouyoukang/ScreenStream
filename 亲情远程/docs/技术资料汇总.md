# 亲情远程 — 技术资料汇总

> **目标**：让远方子女通过浏览器，实时看到并操控父母的安卓手机。
> 零流量中转 · 父母一键可用 · 子女零安装 · 多品牌兼容 · 可验证可交付

---

## 一、技术方案全景对比

### 1.1 屏幕采集技术

| 技术 | 原理 | Root? | ADB? | 延迟 | 画质 | 兼容性 | 适用场景 |
|------|------|-------|------|------|------|--------|---------|
| **MediaProjection API** | Android 5.0+ 官方API，创建VirtualDisplay采集屏幕 | ❌ | ❌ | 50-150ms | ★★★★ | 全品牌 | **首选方案** |
| scrcpy (ADB) | ADB shell 运行server.jar，直接读取framebuffer | ❌ | ✅**始终需要** | 35-70ms | ★★★★★ | 全品牌 | 开发者/技术用户 |
| VNC framebuffer | 读取/dev/graphics/fb0 | ✅ | ❌ | 100-300ms | ★★★ | 旧设备 | 已过时 |
| 虚拟显示器 | VirtualDisplay + MediaCodec | ❌ | ❌ | 30-80ms | ★★★★★ | Android 10+ | 高级方案 |

**结论**：MediaProjection是唯一满足"无root、无ADB、全品牌"要求的采集方案。

### 1.2 输入控制技术

| 技术 | 原理 | Root? | ADB? | 精度 | 兼容性 | 限制 |
|------|------|-------|------|------|--------|------|
| **AccessibilityService** | 系统无障碍API注入手势/键盘事件 | ❌ | ❌ | ★★★★ | 全品牌 | 需用户手动开启 |
| ADB shell input | 通过ADB注入触控/键盘 | ❌ | ✅**始终需要** | ★★★★★ | 全品牌 | ADB连接必须 |
| Shizuku API | 借助ADB权限代理执行特权操作 | ❌ | 一次性 | ★★★★★ | Android 11+ | 需Shizuku常驻 |
| Root inject | 直接操作/dev/input/event* | ✅ | ❌ | ★★★★★ | 有Root设备 | 不可交付 |

**结论**：AccessibilityService是唯一满足要求的控制方案。Shizuku可作为增强（自动启用A11y）。

### 1.3 传输技术

| 技术 | 协议 | 延迟 | P2P? | 浏览器? | 穿透NAT? | 成本 |
|------|------|------|------|---------|---------|------|
| **WebRTC** | UDP/DTLS-SRTP | 50-200ms | ✅ | ✅ | STUN/TURN | ¥0(STUN) |
| MJPEG over HTTP | HTTP长连接 | 100-500ms | ❌ | ✅ | 需穿透 | 依赖带宽 |
| H264 over WebSocket | TCP/WebSocket | 80-300ms | ❌ | ✅ | 需穿透 | 依赖带宽 |
| RTSP | UDP/RTP | 50-150ms | ❌ | ❌(需播放器) | 困难 | ¥0 |
| VNC (RFB) | TCP | 100-500ms | ❌ | ❌(需客户端) | 需穿透 | ¥0 |

**结论**：WebRTC是唯一同时满足"P2P直连+浏览器零安装+NAT穿透"的传输方案。

### 1.4 NAT穿透/公网访问

| 方案 | 原理 | 成本 | 延迟 | 成功率 | 复杂度 |
|------|------|------|------|--------|--------|
| **STUN** | 发现公网IP，尝试P2P直连 | ¥0 | 最低 | 70-85% | 低 |
| **TURN** | 媒体流中继（P2P失败时） | 流量费 | +30-80ms | 100% | 低 |
| **FRP** | 端口穿透到公网服务器 | ¥38/年 | +20-40ms | 100% | 中 |
| Tailscale | WireGuard P2P VPN | ¥0(免费) | 5-20ms(P2P) | 95%+ | 低 |
| Cloudflare Tunnel | HTTP隧道 | ¥0 | 50-150ms | 100% | 低 |

**结论**：WebRTC内置STUN/TURN处理NAT穿透。信令通道用FRP(已部署)或直连。

## 二、开源方案深度分析

### 2.1 ScreenStream（本项目 — ★★★★★ 首选基座）

| 维度 | 现状 |
|------|------|
| **采集** | MediaProjection → MJPEG/H264/H265 + WebRTC P2P |
| **控制** | AccessibilityService, 118+ REST API |
| **传输** | Ktor HTTP Server + WebSocket + WebRTC DataChannel |
| **前端** | 6400行index.html（功能极丰富，但面向技术用户） |
| **公网** | FRP → aiotvr.xyz 已部署（HTTPS反代） |
| **信令** | 自建WebRTC Relay (aiotvr.xyz:9100, systemd) |
| **鉴权** | PIN码 + Bearer Token |
| **P2P** | `WebRtcP2PClient.kt` 已实现（STUN+TURN+DataChannel控制转发） |

**优势**：已构建80%+能力，代码完全可控，无外部依赖。
**差距**：
1. 前端面向技术用户，需要"亲情简化版"
2. WebRTC P2P路径需要端到端验证
3. 信令服务器(`wss://aiotvr.xyz/signal/`)需要确认可用
4. 需要"父母一键启动"模式
5. 缺少品牌适配文档

### 2.2 scrcpy / ws-scrcpy

| 维度 | 评估 |
|------|------|
| **原理** | ADB推送server.jar → 手机端运行 → 编码屏幕 → 通过ADB/TCP传回 |
| **优势** | 延迟极低(35-70ms)，画质最优，控制精准 |
| **致命缺陷** | **始终需要ADB连接**（USB或WiFi ADB），用户必须开启开发者模式 |
| **ws-scrcpy** | Web封装，通过WebSocket桥接，但底层仍需ADB |
| **结论** | ❌ **不可交付**给非技术用户。仅适合开发测试。 |

### 2.3 droidVNC-NG

| 维度 | 评估 |
|------|------|
| **原理** | MediaProjection采集 + VNC(RFB)协议传输 + AccessibilityService控制 |
| **优势** | 无root，Android 7+，开源(Apache-2.0) |
| **缺陷** | VNC协议需要VNC客户端（noVNC可Web化但体验差），无P2P，性能一般 |
| **借鉴** | AccessibilityService的使用方式（本项目已借鉴并超越） |
| **结论** | ❌ 不如ScreenStream。VNC延迟高、需客户端。 |

### 2.4 RustDesk

| 维度 | 评估 |
|------|------|
| **原理** | Rust原生，自研协议，P2P + 自建中继 |
| **优势** | 跨平台，P2P，开源，可自建服务器 |
| **致命缺陷** | **子女端需要安装RustDesk客户端**（桌面/移动APP），不满足"零安装" |
| **结论** | ❌ 不满足"子女端零安装打开浏览器即用"。 |

### 2.5 Headwind Remote

| 维度 | 评估 |
|------|------|
| **原理** | MediaProjection + Janus WebRTC Server + AccessibilityService |
| **优势** | 真正的WebRTC浏览器方案，无需客户端安装 |
| **缺陷** | 需要部署Janus Server(重量级C库)，Docker依赖 |
| **借鉴** | 架构思路（手机App → Janus → 浏览器）可参考 |
| **结论** | ⚠️ 架构优秀但部署重，不如ScreenStream自建WebRTC P2P轻量。 |

## 三、核心技术栈确定

### 3.1 最终技术选型

```
┌─────────────────────────────────────────────────────────┐
│                    亲情远程 技术栈                        │
├─────────────────────────────────────────────────────────┤
│  采集层:  MediaProjection API (Android 5.0+)             │
│  编码层:  MediaCodec H264/H265 硬编码                    │
│  传输层:  WebRTC (P2P, STUN/TURN)                        │
│          + MJPEG/H264 WebSocket (备选/降级)              │
│  控制层:  AccessibilityService (118+ API)                  │
│  信令层:  自建WebSocket信令 (aiotvr.xyz)                 │
│  穿透层:  WebRTC ICE (STUN→P2P, TURN→中继)              │
│          + FRP端口穿透 (备选/保底)                       │
│  前端层:  纯HTML/JS/CSS (零依赖, 浏览器直接打开)          │
│  鉴权层:  房间码 + PIN码 + 时效链接                      │
│  服务端:  aiotvr.xyz (信令+反代, ¥38/年)                 │
└─────────────────────────────────────────────────────────┘
```

### 3.2 数据流架构

```
【父母手机】                    【阿里云】              【子女浏览器】
                                                       
MediaProjection                                        
    ↓                                                  
MediaCodec H264                                        
    ↓                                                  
WebRTC PeerConnection ←──信令──→ 信令服务器 ←──信令──→ WebRTC PeerConnection
    ↓                   (SDP/ICE)  :9100      (SDP/ICE)      ↓
    └──── P2P 直连 (UDP DTLS-SRTP) ──────────────────→ VideoTrack → <video>
                                                       
AccessibilityService ←── DataChannel (双向) ──→ 触控/键盘事件
    ↓                                                  ↑
  Android系统                                     浏览器事件
```

**关键路径**：
- **媒体流 P2P 直连**：手机→浏览器，不经过服务器（零流量中转）
- **信令轻量中转**：仅SDP/ICE交换（几KB，忽略不计）
- **控制命令 P2P**：通过DataChannel，与媒体流共用P2P通道

### 3.3 降级策略

| 优先级 | 方案 | 触发条件 | 用户体验 |
|--------|------|---------|---------|
| P0 | WebRTC P2P直连 | STUN成功打洞 | 最优：低延迟，零中转 |
| P1 | WebRTC TURN中继 | P2P失败，TURN可用 | 良好：延迟+30-80ms |
| P2 | H264 WebSocket via FRP | WebRTC全部失败 | 可用：延迟100-300ms |
| P3 | MJPEG via FRP | 极端情况 | 基本可用：延迟较高 |

## 五、品牌适配关键问题

### 5.1 MediaProjection 品牌差异

| 品牌 | 已知问题 | 解决方案 |
|------|---------|---------|
| **华为/荣耀** | EMUI限制后台投屏，弹二次确认 | 加入电池优化白名单，关闭智能省电 |
| **小米** | MIUI后台限制严格，可能杀后台 | 加入自启动白名单，锁定后台 |
| **OPPO/realme** | ColorOS电池管理激进 | 加入省电白名单，关闭应用冻结 |
| **vivo/iQOO** | OriginOS后台策略 | 允许后台高耗电 |
| **三星** | OneUI电池优化 | 关闭优化该应用 |
| **OnePlus** | OxygenOS较宽松 | 默认即可（已测试环境） |

### 5.2 AccessibilityService 品牌差异

| 品牌 | 已知问题 | 解决方案 |
|------|---------|---------|
| **华为/荣耀** | 系统可能定期关闭无障碍服务 | 引导用户开启"允许受保护的应用" |
| **小米** | MIUI可能弹窗确认 | 引导截图+步骤说明 |
| **OPPO** | 无障碍设置路径不同 | 品牌特异化引导页 |
| **所有品牌** | Android 13+ 限制无障碍权限 | Shizuku辅助（一次性ADB授权） |

### 5.3 后台保活策略（通用）

```
1. 前台服务 (ForegroundService) — 显示通知，防止系统杀
2. WakeLock — 防止CPU休眠
3. 电池优化白名单 — 请求用户排除
4. 自启动白名单 — 品牌特异化引导
5. 锁定任务 — 防止用户误清
```
