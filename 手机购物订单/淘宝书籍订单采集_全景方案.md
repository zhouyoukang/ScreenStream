## 三、祸之识别与根治

### 祸一：WebView不可读

| 维度 | 详情 |
|------|------|
| **现象** | 淘宝订单详情页用WebView渲染，`uiautomator dump`只能获取容器节点，无法读取HTML内文本 |
| **受害五感** | 手机👁(View树残缺) → Agent👁(看不到详情) → 用户👅(数据不完整) |
| **根因** | Android AccessibilityService对WebView的支持有限，`uiautomator`跨进程读不到WebView DOM |
| **根治** | **ScreenStream的AccessibilityService可以读取WebView内容**——通过`/screen/text`而非`uiautomator dump`。SS的InputService绑定到rootInActiveWindow，对WebView有更好的覆盖（FINDINGS.md P22确认：WebView返回DOM ID非文本，但text字段仍可获取部分内容）。如果SS也读不到，降级方案：截图+OCR(ADB screencap → PC端pytesseract) |
| **实施** | 优先用`phone_lib.read()` (`/screen/text`)；降级用`/viewtree?depth=8&webview=true`；最终降级用截图OCR |

### 祸二：订单卡片需双击/特殊交互进入详情

| 维度 | 详情 |
|------|------|
| **现象** | 淘宝订单列表中，单击订单卡片可能不进入详情页（需要点击商品图片/商品名区域） |
| **受害五感** | Agent🖐(点错位置) → Agent👁(停留在列表页以为进入了详情) |
| **根因** | 淘宝订单卡片的可点击区域是商品图片和商品名，不是店铺名或状态栏 |
| **根治** | 用`/viewtree?depth=6`精确定位`clickable=true`的商品名节点→取其bounds中心坐标→`/tap`点击。验证：点击后检查`/foreground`是否变为`OrderDetailActivity`或URL变化 |
| **实施** | 遍历viewtree找到长文本(>15字符)+clickable=true+y在有效区域的节点→tap其中心 |

### 祸三：滚动到底误判

| 维度 | 详情 |
|------|------|
| **现象** | 上次采集只滚动3屏就认为到底(实际有14+笔)。连续空屏阈值过低 |
| **受害五感** | Agent👁(以为到底了) → 用户👅(数据严重不全) |
| **根因** | 1)淘宝加载慢，滑动后内容未渲染→空屏误判 2)签名对比过于严格 3)淘宝可能有"加载更多"按钮而非无限滚动 |
| **根治** | 1)空屏阈值从3→5次 2)每次滑动后等待2秒而非1.5秒 3)检测"加载更多"/"查看更多"文本并点击 4)检测"没有更多了"/"已经到底了"作为真正终止信号 5)计数总订单数与屏幕显示的总数对比 |
| **实施** | `taobao_deep_collect.py`已有max_scrolls=50和3次空屏退出。改为5次空屏+检测"没有更多"文本 |

### 祸四：淘宝登录态过期

| 维度 | 详情 |
|------|------|
| **现象** | 淘宝APP长时间未使用可能需要重新登录，或进入订单页弹出登录框 |
| **受害五感** | Agent🖐(操作被登录框阻断) → 所有感官链断裂 |
| **根因** | 淘宝安全策略，长期未活跃的账号需要重新验证 |
| **根治** | 1)采集前先检测：打开淘宝→读屏→检查是否有"登录"/"验证"文本 2)如果需要登录→**这是唯一需要用户介入的点**（验证码/人脸无法自动化） 3)登录完成后Agent继续 |
| **实施** | 脚本开头加login_check()：读屏→搜索"登录"/"验证"→有则暂停提示用户→`/wait?text=全部订单&timeout=120` |

### 祸五：OPPO安全弹窗拦截

## 四、惑之解答

### 惑一：纯ADB vs ScreenStream API，用哪个？

**答**: **混合模式最优**。

| 操作 | 用ADB | 用SS API | 推荐 |
|------|-------|----------|------|
| 启动APP | `monkey`(绕过弹窗) | `/openapp`(可能被拦截) | ADB |
| 读取屏幕 | `uiautomator dump`(200-500ms) | `/screen/text`(10-30ms) | **SS API** |
| 点击元素 | `input tap x y` | `/findclick`(语义+坐标) | **SS API** |
| 滑动翻页 | `input swipe`(精确) | `/swipe`(归一化) | 均可 |
| 读WebView | ❌不可用 | 部分可读 | **SS API** |
| 读详情页 | `uiautomator dump` | `/screen/text`+`/viewtree` | **SS API** |

### 惑二：列表页扫描 vs 逐单进详情，选哪个？

**答**: **两阶段采集**。

1. **Phase A: 列表页全量滚动** — 快速获取所有订单的商品名+价格+店铺+状态
2. **Phase B: 书籍订单钻入详情** — 仅对Phase A识别为书籍的订单进入详情页，获取订单号+日期+实付款

Phase A约3-5分钟(50屏)，Phase B约每单15-20秒。假设20笔书籍订单→Phase B约5-7分钟。总计约10分钟。

### 惑三：如何确保不遗漏？

**答**: 三重保障。

1. **滚动完整性**：检测"没有更多了"文本 + 连续5次空屏 + 总数与页面显示对比
2. **跨屏去重**：以(店铺名+商品名前20字)为唯一键
3. **交叉验证**：列表页数据 vs 详情页数据 取并集

### 惑四：用户需要做什么？

**答**: **理想情况下零操作**。唯一例外：
- 如果淘宝需要登录/验证 → 用户在手机上完成登录 → Agent自动继续
- 如果MediaProjection弹窗 → ScreenStream的投屏授权(通常启动时已授权)

### 惑五：多平台书籍如何汇总？

**答**: `scan_book_orders.py`已覆盖7平台(当当→京东→淘宝→拼多多→闲鱼→转转→亚马逊)。本次核心聚焦淘宝，其他平台作为bonus。

从已有数据看，**拼多多也有大量教材订单**：
- 高等数学第八版(上/下) ¥14.59/¥13.90
- 管理学第二版 ¥13.77/¥14.00
- 材料概论(6笔不同卖家) ¥9.31~¥19.90
- 线性代数第三版 ¥28.78/¥29.70
- 旅游接待业 ¥3.83
- 光催化 ¥21.78
- 机械制造基础实习 ¥4.80
