# 远程协助投屏 — 技术路径全景研究 v2

> **道**：让每一位不擅数字设备的长辈，都能通过家人的远程"在场"，像被人牵着手一样自如地使用手机。
> **约束**：免Root、免ADB、国内网络、Android 8.0+、品牌无关、中老年可独立完成、30秒连通。
> **更新**: 2026-03-01 (v2, 从10条扩展至15条路径, 新增厂商内置/TeamViewer/AirDroid/微信小程序/Telegram方向)

## 二、存活方案深度分析

### 🥇 方案1：厂商内置远程协助（零门槛标杆）

> **这是唯一真正零门槛的方案**——系统预装，无需安装任何APP，无需开发者选项。

| 品牌 | 功能名称 | 入口 | 跨品牌 | 反控 | 当前状态 |
|------|---------|------|--------|------|---------|
| **OPPO** (ColorOS 11+) | 屏幕共享 | 设置→连接与共享→屏幕共享 | ❌ 仅OPPO→OPPO | ✅ 授权后可控 | ✅ 可用 |
| **华为** | 畅连通话·屏幕共享 | 畅连APP→视频通话→共享屏幕 | ⚠️ 华为设备间 | ✅ 可涂鸦标注 | ⚠️ 畅连APP仍在但功能缩减 |
| **小米** | 小米通话·远程协助 | 小米通话APP→远程协助 | ❌ 仅小米→小米 | ✅ 授权后可控 | ✅ 可用(原米聊迁移) |
| **vivo** | 远程协助 | vivo官网APP→远程协助 | ❌ 仅vivo→vivo | ✅ 授权后可控 | ✅ 可用 |

**致命缺陷**：**仅限同品牌**。父母用OPPO、孩子用iPhone/小米 → 完全不可用。
**价值**：作为**同品牌场景的首选推荐**，零门槛无可匹敌；跨品牌场景需其他方案。

**操作流程（以OPPO为例）**：
1. 被控端(老人)：设置 → 连接与共享 → 屏幕共享 → 发起共享 → 输入对方手机号 → 发起邀请
2. 控制端(家人)：收到邀请 → 接受 → 看到屏幕
3. 被控端：点"授权对方操作" → 家人可操控

### 🥈 方案2：原生WebRTC（ScreenStream P2P）— 推荐主力方案

> **家人端零安装**（浏览器即可）+ **P2P直连**（最低延迟）+ **全品牌兼容**

**原理**：ScreenStream APP 用 `stream-webrtc-android` 库，MediaProjection → VideoTrack → PeerConnection → P2P直连浏览器。

- **优势**：
  - 家人只需打开浏览器URL，**零安装**
  - P2P直连，服务器零带宽，延迟最低(30-100ms)
  - 已有90%代码：MediaProjection + InputService + CloudRelay + 信令服务器
  - 成本零：免费STUN(Google/小米/B站) + 免费TURN(metered.ca) + 自有ECS信令
- **劣势**：
  - 需开发Kotlin WebRTC集成(1-2天)
  - 老人需安装APP+开无障碍(一次性)
  - WebRTC NAT穿透率80-95%，不如商业方案99%
- **降级链**：P2P STUN → TURN中转 → CloudRelay WS兜底（用户无感知）

### 🥉 方案3：RustDesk（成熟开源，改造复用）

> 68K★ GitHub，最成熟的开源远控方案。

- **优势**：P2P+TURN双路径、全平台、成熟稳定、社区活跃
- **劣势**：**双端都要装APP**（家人也要装RustDesk客户端）、UI非中老年友好
- **自建成本**：阿里云ECS部署hbbs/hbbr，已有服务器可复用
- **操作**：安装APP → 开无障碍 → 读9位ID → 告知家人 → 等连接

### 方案4：TeamViewer QuickSupport（商业免费）

> 老牌商业远控，个人使用免费。

- **优势**：最稳定的商业基础设施、99%穿透率、品牌适配好
- **劣势**：
  - 免费版有"疑似商业使用"检测，频繁使用可能被封
  - 需安装QuickSupport APP + Accessibility Add-on(部分品牌)
  - 数据经TeamViewer服务器（隐私顾虑）
- **操作**：安装QuickSupport → 开无障碍 → 读9位ID → 告知家人 → 等连接

### 方案5：AirDroid Remote Support

> 国产商业方案，有免费基础版。

- **优势**：国内网络优化好、中文界面、支持浏览器控制端
- **劣势**：需安装AirDroid APP + Control Add-on插件 + 开无障碍（三步）、免费版有限制
- **操作**：安装AirDroid+插件 → 开无障碍 → 分享9位码 → 等连接

### 方案6：ScreenStream CloudRelay（已部署可用）

> 当前已部署运行的方案，手机APP一键投屏到公网浏览器。

- **优势**：已经部署运行✅、家人浏览器即可、自有服务器(数据可控)
- **劣势**：视频经服务器中转(带宽受限，阿里云2核2G)、非P2P
- **当前状态**：aiotvr.xyz/cast/ 已可用，E2E验证通过(270×602 17fps)

### 方案7：向日葵/ToDesk（商业付费）

- 免费版：限速1Mbps(向日葵)/限时2小时(ToDesk)、有广告
- 付费版：¥100-200/年
- 操作：双端装APP → 注册账号 → 开无障碍 → 输入识别码 → 连接

### 方案8：腾讯应用宝远程协助

- **极大优势**：微信分享链接即可连接，老人最熟悉微信
- **致命劣势**：仅支持部分手机（品牌/型号限制多）、功能已多年未更新、可能随时下线

## 三、淘汰路径详细记录

### 路径9: scrcpy over network
- **验证**: 文档+实测确认
- **致命点**: 必须启用USB调试(开发者选项→USB调试→PC授权)，老人完全不可能操作
- **教训**: 任何依赖ADB的方案对非技术用户都是死路

### 路径10: Miracast/Android Cast
- **验证**: 协议文档确认
- **致命点**: WiFi Direct仅局域网(同一WiFi)；无反向操控能力；部分品牌支持不完整
- **教训**: 局域网协议天然不满足公网需求

### 路径11: 云手机 (红手指/多多云)
- **验证**: 产品分析
- **致命点**: 控制的是云端虚拟Android，不是老人手里的真实手机。无法帮老人操作微信/设置。
- **教训**: 需求是"看到+操控老人的真实手机"，而非"有一台远程手机"

### 路径12: Tailscale/ZeroTier + VNC
- **验证**: 部署测试
- **致命点**: 需4步配置(注册→装VPN APP→连网络→装VNC)，每步都有技术概念(虚拟网络/VNC协议)
- **教训**: 步骤数>3的技术方案对老人都不现实

### 路径13: PeerJS浏览器桥接
- **验证**: 2026-03-01 实机测试（OnePlus NE2210）
- **致命点**: 两个架构级缺陷构成死锁：
  1. Android将后台Chrome的WebSocket挂起 → bridge切后台则0fps
  2. Chrome bridge在前台时，MediaProjection捕获Chrome自身UI → 反馈环
- **教训**: 浏览器不是后台常驻进程的合适宿主；需原生前台Service

### 路径14: 微信小程序投屏
- **验证**: 微信小程序API文档确认
- **致命点**: 小程序沙箱无法调用MediaProjection API，无法捕获屏幕内容。微信本身也无开放投屏SDK。
- **教训**: 微信小程序的系统权限极其受限，不适合做系统级投屏

### 路径15: Telegram Bot截图投屏
- **验证**: 方案评估
- **致命点**: 只能逐帧截图→上传→下载，延迟>5秒/帧，无法实现实时投屏；无反向操控通道
- **教训**: 消息平台适合通知/命令，不适合实时媒体流

## 四、推荐方案及降级策略

### 🏆 分场景推荐（而非单一方案）

> **核心洞察**：没有银弹。不同家庭场景需要不同方案。正确策略是**按场景匹配**。

#### 场景A: 父母和孩子同品牌手机
```
首选: 厂商内置远程协助 (方案1)
理由: 零安装、零配置、电话指导即可完成、最低认知负担
```

#### 场景B: 跨品牌 / 孩子用iPhone / 孩子要用电脑帮忙
```
首选: 原生WebRTC P2P (方案2) + CloudRelay降级 (方案6)
理由: 家人端零安装(浏览器即可)、P2P最低延迟、自有基础设施、完全可控
备选: RustDesk (方案3) — 如果愿意双端装APP
```

#### 场景C: 不想折腾、立即可用
```
首选: ScreenStream CloudRelay (方案6) — 已部署运行
理由: 现在就能用、APK已编译、公网页面已部署
备选: TeamViewer QuickSupport (方案4) — 最稳定的商业免费方案
```

### 三级降级策略（原生WebRTC方案内部）

```
Level 1: WebRTC P2P直连 (STUN穿透成功)
         → 延迟30-80ms, 零服务器带宽, 最优体验
         → 适用: ~80%家庭网络(锥形NAT/端口限制NAT)

Level 2: WebRTC TURN中转 (对称NAT, STUN失败)
         → 延迟80-150ms, TURN带宽, 仍为WebRTC流
         → 适用: ~15%家庭网络(对称NAT/运营商级NAT)
         → 免费TURN: metered.ca (全球CDN, 500MB/月)

Level 3: CloudRelay WS中转 (WebRTC完全失败)
         → 延迟~50ms, 自有ECS带宽, H264 WS帧转发
         → 适用: 极端网络环境(企业防火墙/全阻断)
         → 已部署: aiotvr.xyz relay :9800
```

**用户感知**: 只看到"已连接 ✅"，不知道走的哪条路径。路径切换完全自动、无感知。

### 从其他方案借鉴的精华

| 来源 | 借鉴点 | 用于 |
|------|--------|------|
| RustDesk | 无障碍权限引导流程(带截图分步) | APP首次启动引导 |
| RustDesk | 9位数字设备ID | 比6位更安全 |
| 厂商方案 | 手机号配对(OPPO) | 未来：微信好友配对 |
| TeamViewer | 一次性会话密码 | 每次共享生成新密码 |
| 五感审计 | 品牌特有省电/自启陷阱检测 | 安装后自动检测+引导 |

## 七、安全设计

| 威胁 | 防护 |

## 九、核心洞察

> **技术不是门槛，恐惧才是。**
>
> 65岁的老人不是不会点屏幕——他们每天在微信上发语音、看视频、抢红包。
> 让他们害怕的不是"操作"，而是"未知"：
> - 看到英文就害怕
> - 看到"此应用将监控您的操作"就害怕
> - 看到手机自己在动就害怕
>
> 我们的工作不是造一个"功能完善的投屏工具"，
> 而是造一个**"不需要害怕的工具"**。
>
> **设计原则**：
> 1. **一个屏幕一个动作** — 永远不让老人面对选择
> 2. **零术语** — "让家人看我的屏幕"而非"开始投屏"
> 3. **预防恐惧** — 每个系统弹窗前先用温暖文案打底
> 4. **自动一切** — 编码/分辨率/网络/重连全自动
> 5. **随时可停** — 停止按钮无处不在，停了就断了
