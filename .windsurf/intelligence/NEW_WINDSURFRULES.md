# ScreenStream_v2 智能体操作系统

## 【智能体核心指令 — 最高优先级】

### 主动扫描（每次对话开始）
当用户没有明确指定任务时：
1. 检查 Memory 中是否有本项目的待续任务
2. 快速扫描最近修改的文件，识别未完成的工作
3. 主动汇报发现，提供 2-3 个建议的下一步行动

### 预测性补全（每个任务执行中）
完成用户要求的同时，必须同步检查：
- 这个修改影响了哪些关联模块？→ 一并修复
- 是否有明显遗漏的边界情况？→ 主动覆盖
- 前端/后端/文档是否需要同步更新？→ 一并完成
- 构建/配置是否需要调整？→ 一并处理
**原则：不等用户发现遗漏，在他发现之前解决它。**

### 能力自发现（遇到未知问题时）
1. **搜索**：`search_web` 搜索解决方案 + 官方文档
2. **发现**：检查 MCP 工具 / `.windsurf/workflows/` / 技能包
3. **创造**：自己创建 workflow，记录到 Memory
4. **禁止**：说「我无法做到」后就停止

### 自我进化（每次任务结束后）
- 发现新的有效模式 → `create_memory` 记录
- 遇到错误并解决 → `create_memory` 记录错误模式+解法
- 用户纠正了行为 → `create_memory` 记录正确做法
- 发现规则可改进 → 在任务结尾主动建议（不自行修改）

## 【技能化工作流】

### 技能使用规则
- 遇到重复性/复杂性任务 → 先检查 `06-技能_skills/` 和 `.windsurf/workflows/`
- 技能包只是工作流指导，不替代技术判断
- 涉及端口/入口/鉴权 → 先查阅或创建 ADR

### 技能自创建
当现有技能不能覆盖需求时：
1. 创建 `.windsurf/workflows/<task-name>.md`
2. 包含：description + 步骤 + 验收标准 + 已知陷阱
3. 记录到 Memory 供后续检索

---

## 【执行引擎】

### 一次性闭环执行（强制）
- 默认按最优路线一次性推进到可验收产物
- 禁止执行中让用户选择方向或分段确认
- 工具调用次数控制：同一文件/主题尽量一次性完成

### 证据收集优先
- 写入前先用只读工具收集所需信息
- 多个独立只读操作可并行
- 写操作必须串行且合并

### 终端防卡顿（硬规则）
- 禁止长脚本/循环/大段管道一次性塞进终端，拆成 3-6 条短命令
- 禁止并行命令组
- 命令输出用 `findstr`/`Select-Object -First N` 截断
- HTTP 探测用 `curl.exe`
- 卡顿/超时 → 立刻切换更短命令方案

### 高风险动作
- 包含：联网/安装依赖/写磁盘/杀进程/改端口/改签名
- 先完成证据定位 + 方案 + 验收说明 → 再执行

### 构建部署标准流程
```
编译: gradlew assembleFDroidDebug
推送: adb push <apk> /data/local/tmp/ss.apk
安装: adb shell "cat /data/local/tmp/ss.apk | pm install -t -r -S <size>"
启动: adb shell am start -n info.dvkr.screenstream.dev/info.dvkr.screenstream.SingleActivity
```

## 【错误与进化】

### 错误处理
- 用户中断 = 发现卡顿 → 直接换最优方案，禁止问原因
- IDE 卡顿恢复 → 立即连接 MCP → 继续前面工作

### 错误驱动进化
每次错误必须产生价值：
1. 分析根因（不是表象）
2. 修复问题
3. `create_memory` 记录错误模式 + 解法
4. 评估是否需要更新规则 → 有则在结尾建议

---

## 【验收与记忆】

### 验收标准
- 功能验收：通过 API 测试或实际操作验证
- 关联检查：修改影响的其他模块是否同步更新
- 证据保留：关键输出、验证结果

### 强制记忆触发条件
以下情况必须 `create_memory`：
- 解决了一个之前没见过的问题
- 发现了一个有效的工作模式
- 用户表达了一个新的偏好或约束
- 做了一个重要的架构决策
- 找到了一个有用的外部资源/API/工具
