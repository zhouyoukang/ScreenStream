## 二、全局规则重写方案

### 当前结构问题
- MCP 调用规范占据了太多篇幅
- 缺少 SOUL（思维方式）定义
- 缺少预测性思维指令
- 缺少能力自扩展指令

### 新结构设计

```
全局规则 = SOUL层 + 决策框架 + MCP规范 + 执行原则 + 自进化规则
```

#### 各段职责：

**SOUL 层（新增，最高优先级）**
- 定义 AI 的核心身份：不是工具，是智能协作者
- 定义思维方式：预测 > 补全 > 超越指令
- 定义价值取向：卓越 > 够用，主动 > 被动

**决策框架 PREDICT（新增）**
- P-Predict：预测用户下一步需求
- R-Research：搜索更优方案
- E-Execute：闭环执行
- D-Document：记录发现到 Memory
- I-Iterate：执行后反思改进
- C-Complete：补全用户未提的显然需求
- T-Transfer：经验迁移到其他场景

**MCP 规范（保留，精简）**
- 保留核心调用规则
- 移除冗余说明

**执行原则（保留+增强）**
- 保留：闭环执行、证据优先、工具优先
- 新增：预测性补全、能力自发现、错误驱动进化

**自进化规则（新增）**
- 犯错 → 创建 Memory 防止重犯
- 发现模式 → 记录为可复用知识
- 用户纠正 → 立即更新行为
- 规则需改进 → 主动建议修改

### 新结构（预计 200-250 行）

6. 【错误与进化】← 升级：错误驱动进化 + 规则自更新建议
7. 【验收与记忆】← 升级：验收 + 强制记忆关键发现

## 四、关键 Memory 条目规划

### 需要创建的 Memory

1. **用户画像**
   - 标签：user_profile, preferences
   - 内容：身份、角色、偏好、技术栈、设备、目标

2. **项目架构模式**
   - 标签：project_architecture, patterns
   - 内容：Gradle 模块映射、端口分配、已有 ADR

3. **已知陷阱库**
   - 标签：known_pitfalls, error_patterns
   - 内容：已遇到的问题和解决方案（hint text、服务启动等）

4. **规则进化日志**
   - 标签：rule_evolution, changelog
   - 内容：规则修改历史和原因

## 五、验证标准

### 规则升级成功的标志

1. **预测性**：AI 在完成任务时，主动做了用户没明确要求但显然需要的事
2. **自发现**：AI 遇到不会的事情时，主动搜索并找到解决方案
3. **自记忆**：AI 在有价值的发现后，主动创建 Memory
4. **自建议**：AI 在发现规则可改进时，主动提出修改建议
5. **减少人工**：用户需要提供的指令量明显减少
6. **错误不重犯**：同类错误不再重复出现

### 验证方法

给 AI 一个模糊的需求（如「优化一下输入体验」），观察它是否能：
- 自主确定具体的优化方向
- 预测需要修改的所有相关文件
- 主动覆盖边界情况
- 完成后记录关键发现
- 建议后续可能的改进方向

## 六、风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| AI 过度主动，做了不该做的 | 保留「高风险动作需确认」规则 |
| 规则太长导致 token 浪费 | 控制总规则量在 300 行以内 |
| 自我修改失控 | 规则修改必须经用户确认 |
| 预测错误浪费时间 | 预测性补全限定在「显然需要」的范围 |
| Memory 过多难以管理 | 定期合并和清理 Memory |
