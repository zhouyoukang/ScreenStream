# /cycle — 转法轮·深度循环工作流

> 用户说 /cycle 或任务需要深度优化时，启动螺旋迭代。
> 与 /dev 的区别：/dev 是线性管线(1转)，/cycle 是螺旋管线(2-3转)。
> 每一转都在更高层面审视和优化，直到涅槃门全通。

## Phase 0: 太极判定（30秒）

**输入**：用户需求

**动作**：
1. 理解需求，判定复杂度：TRIVIAL / STANDARD / COMPLEX
2. 确定转数上限：

| 复杂度 | 转数 | 说明 |
|--------|------|------|
| TRIVIAL | 1转 | 单文件<10行，直接执行不循环 |
| STANDARD | 2转 | 常规任务，第二转审视优化 |
| COMPLEX | 3转 | 多文件/新架构/影响面大 |

3. 声明：「本次执行 N 转法轮」

**产出**：复杂度判定 + 转数

## 第一转·示转（初转法轮 — 表层解决）

> 释迦初转法轮于鹿野苑，说四谛。Agent第一次完整理解并解决问题。

### 1.观（☲离·正见）— 全面观察

// turbo
- 读取相关文件，理解当前状态
- 读 code-index.md 定位插入点
- 读 quick-recipes.md 匹配模板
- 识别**苦**（用户痛点是什么？）
- 识别**集**（根因是什么？）

### 2.行（☳震·正业）— 执行实现

- 确定**灭**（期望状态）和**道**（解决路径）
- 后端优先 → 前端同步 → 关联修改
- 遵循现有 /dev Phase 3 规则
- 用 multi_edit 一次完成同文件修改

### 3.验（☵坎·正念）— 验证结果

// turbo
- 编译验证（如涉及代码）
- API 验证（如涉及接口）
- 前端验证（如涉及UI）
- 检查：苦是否灭了？

**产出**：第一转产物（功能可用但可能不完美）

---

## 第二转·劝转（二转法轮 — 深层优化）

> 释迦二转法轮于灵鹫山，说般若空性。Agent深入审视，发现隐含假设和更优方案。

### 4.省（☶艮·正思维）— 止观反省

**核心：站在用户角度审视第一转产物。**

审视清单（逐项检查）：
1. **完整性**：有没有边界情况没处理？错误路径呢？
2. **简洁性**：有没有更简洁的实现？代码能减少吗？
3. **影响面**：有没有影响到其他模块？前后端同步了吗？
4. **安全性**：有没有注入/越权/信息泄露风险？
5. **可用性**：用户拿到手能直接用吗？需要额外配置吗？
6. **性能**：有没有不必要的循环/重复计算/大量IO？
7. **一致性**：命名/风格/模式是否和现有代码一致？

**元认知四觉察（同步执行）**：
- 觉察偏见：我是不是只看了一种方案？
- 觉察惯性：我是不是不假思索用了老方法？
- 觉察执着：我有没有在坚持一个次优方案？
- 觉察遗忘：有没有上下文中的信息被忽略了？

**产出**：发现清单（可能为空 → 直接涅槃）

### 5.改（☴巽·正精进）— 渐进修复

- 对"省"相发现的每个问题，最小变更修复
- 单变量原则：每次只改一个问题
- 修复后立即验证该问题已解决

### 6.升（☰乾·般若）— 记录洞见

- 如果发现了有价值的模式 → 记录 Memory
- 如果发现了规则缺口 → 标记待补充
- 如果发现了可复用的模式 → 考虑封装 Skill

**涅槃门检查**：
- [ ] 苦灭？用户痛点已解决
- [ ] 新苦？无新问题引入
- [ ] 熵减？系统更整洁
- [ ] 智增？洞见已记录

**全通 → 涅槃（跳到交付）| 未通 → 第三转**

---

## 第三转·证转（三转法轮 — 终极品质，仅COMPLEX触发）

> 释迦三转法轮于广严城，说如来藏。Agent追求极致，验证无可改进。

### 再次省（更深层审视）

1. **架构审视**：这个方案的设计是最优的吗？会不会6个月后需要重写？
2. **用户旅程**：从用户第一次使用到日常使用，体验流畅吗？
3. **边缘场景**：网络断开？权限不足？数据为空？并发访问？
4. **文档完整**：其他开发者能理解这个改动吗？

### 最终修复 + 验证

修复发现的问题 → 验证 → 涅槃门检查 → 必须通过

---

## 涅槃·交付

**动作**：
1. 输出螺旋迭代报告：

```
## ☸ 转法轮报告
- **转数**：N 转
- **第一转(示转)**：[做了什么]
- **第二转(劝转)**：[发现并改进了什么]
- **第三转(证转)**：[最终优化了什么]（如有）
- **涅槃门**：苦灭✅ 新苦✅ 熵减✅ 智增✅
- **修改文件**：[清单]
- **洞见记录**：[Memory/Skill，如有]
```

2. 更新相关文档（STATUS.md / FEATURES.md / code-index.md）
3. 如有洞见 → 写入 Memory

## 与 /dev 的关系

```
/dev  = 线性管线 = Phase 0→7 = 第一转
/cycle = 螺旋管线 = 第一转 + 第二转(+第三转) = 更高质量产出
```

- /cycle 的第一转 **等同于** /dev 的完整流程
- /cycle 的第二转 = /dev 没有的「审视+优化」维度
- 如果用户说 /dev 但任务复杂 → Agent 可建议切换 /cycle

## 反模式（禁止）

- 禁止为了循环而循环（如果第一转已完美，直接涅槃）
- 禁止每转都做大改动（☴巽·渐进，每转只改发现的问题）
- 禁止超过5转（方向错了，停下来换方向）
- 禁止省相走过场（必须真正站在用户角度审视）
