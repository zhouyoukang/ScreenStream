# 自我进化工作流

当用户说 /evolve 或 AI 在工作中积累了足够的观察时，按以下步骤执行系统性自我评估和进化。

## Phase 1: 规则审查

// turbo
读取所有 `.windsurf/rules/*.md` 文件，检查：
- 有没有**过时的规则**（描述的问题已不存在、技术已更换）？
- 有没有**冲突的规则**（A 规则说做 X，B 规则说不做 X）？
- 有没有**缺失的规则**（最近遇到的场景没有规则覆盖）？
- 有没有**臃肿的规则**（太长、太细、重复描述同一件事）？

输出：规则健康度评分 + 具体改进建议

## Phase 2: Skills 审查

// turbo
读取所有 `.windsurf/skills/*/SKILL.md`，检查：
- 现有 Skills 是否都还有效？步骤是否需要更新？
- 有没有**重复执行的操作**应该被封装为新 Skill？
- 全局 Skills（~/.codeium/windsurf/skills/）有没有可以利用但还没用到的？

输出：Skills 覆盖度评估 + 建议新增/更新的 Skills

## Phase 3: Memory 审查

搜索最近的 Memory 条目：
- 有没有**过期的记忆**需要更新？（技术已变更、问题已解决）
- 有没有**重复的记忆**可以合并？
- 有没有**缺失的记忆**？（重要的项目知识没有记录）

输出：Memory 健康度 + 清理/补充建议

## Phase 4: 工具与 MCP 审查

评估当前可用的 MCP 服务和工具：
- 当前 MCP 服务（playwright/chrome-devtools/context7/github，共4个核心）使用情况如何？
- 有没有**未充分利用**的 MCP 能力？（如 `browser_run_code` token降99%）
- 有没有**新的 MCP 服务**值得建议用户添加？（参考 `文档/BROWSER_MCP_MULTI_AGENT_RESEARCH.md` §2.4）
- 上下文税是否合理？（当前4个Server约11-14%，参考Bug0 2026实测）
- IDE 配置是否最优？

输出：工具能力评估 + 提升建议

## Phase 5: 用户交互模式分析

基于最近几次对话的 Memory 记录：
- 用户最常做什么类型的任务？规则是否充分支持？
- 用户的提示词质量趋势？有没有改善或退步？
- 用户倾向于什么样的交互风格？规则是否匹配？
- 有没有用户反复遇到但规则没有处理的情况？

输出：用户画像更新 + 交互优化建议

## Phase 6: 实施进化

根据 Phase 1-5 的发现，按以下优先级执行：

1. **立即修复**：明确的规则错误、过期信息 → 直接修改
2. **建议修改**：需要权衡的改进 → 列出方案让用户选择
3. **记录观察**：暂时不确定是否需要改的 → 写入 Memory 继续观察
4. **长期规划**：需要较大改动的 → 记录为下次 /evolve 的待办
