# /optimize — 性能优化工作流

## Phase 1: 定位瓶颈

1. **时间复杂度分析** — 找出 O(n²) 或更差的算法
2. **I/O 瓶颈** — 文件/网络/数据库调用频率
3. **内存分析** — 大对象分配、泄漏风险
4. **热点函数** — 被频繁调用的代码路径

工具：
```bash
# Android/JVM
adb shell dumpsys meminfo <package>

# Node.js
node --prof app.js && node --prof-process isolate-*.log

# Python
python -m cProfile -s cumulative script.py
```

## Phase 2: 选择优化策略

| 瓶颈类型 | 优化策略 | 风险 |
|---------|---------|------|
| 算法 O(n²) | 换更优数据结构/算法 | 低 |
| 重复计算 | 缓存/记忆化 | 低 |
| 多次小I/O | 批处理合并 | 中 |
| 同步阻塞 | 异步/并发 | 中 |
| 大对象频繁分配 | 对象池/复用 | 中 |
| 不必要的加载 | 惰性加载/按需初始化 | 低 |

## Phase 3: 实施优化

**原则**：
- 先测量，再优化，再测量
- 每次只改一个瓶颈点
- 不以牺牲可读性为代价
- 保留优化前的性能数据作为对比基线

## Phase 4: 验证

1. 对比优化前后的指标
2. 确认优化没有引入功能回归
3. 文档化优化决策和效果

```
## 优化结果
| 指标 | 优化前 | 优化后 | 提升 |
|------|-------|-------|------|
| [指标] | [值] | [值] | [%] |
```
