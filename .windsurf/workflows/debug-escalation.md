# Bug 调试升级工作流

## 适用场景
- Bug 原因不明确
- 涉及 IDE/系统/全局配置问题
- 第一次尝试修复未成功

## 流程

### Phase 1: 冻结现场（必须先做）
1. **记录当前状态**：列出所有相关文件的当前内容（用 `read_file`）
2. **记录症状**：精确描述问题现象（什么操作→什么结果→预期是什么）
3. **建立回滚点**：对即将修改的文件先备份（`.backup_日期` 后缀）

### Phase 2: L1 自主分析（最多2轮）
1. 根据已有知识分析可能原因
2. **单变量修改**：每次只改一个配置/文件，确认效果后再改下一个
3. 请用户确认修改后的实际效果（AI 看到的 ≠ 用户看到的）
4. 如果 2 轮未解决 → 立即进入 Phase 3，**禁止继续猜测**

### Phase 3: L2 资料引入
1. `search_web` 搜索：官方文档 + GitHub Issues + Reddit/社区讨论
2. 查阅 Windsurf 官方文档 `docs.windsurf.com`
3. 对比搜索结果与当前症状，找到匹配的已知问题
4. 如果找到方案 → 按方案执行；如果未找到 → 进入 Phase 4

### Phase 4: L3 人机协作
向用户发起结构化求助，必须包含：
```
## 当前问题
[精确描述症状]

## 已尝试
1. [方案1] → [结果]
2. [方案2] → [结果]

## 我需要你帮忙的
- [ ] [具体操作1]：因为 [AI为什么做不了]
- [ ] [具体观察1]：请告诉我 [你看到了什么]

## 我的猜测方向
A. [可能原因1]
B. [可能原因2]
```

### Phase 5: L4 用户指挥
用户提供方向性指导后，AI 负责：
1. 按用户指示的方向深入调查
2. 执行具体技术操作
3. 汇报结果，请用户判断是否解决

## 关键纪律
- **禁止在 Phase 2 超过 2 轮**：超过就是在浪费对话且可能引入新问题
- **每次修改前备份**：确保任何时候都能一键回滚
- **AI 成功 ≠ 真成功**：必须由用户确认实际效果
- **恢复优先于重建**：如果修改导致新问题，先回滚到备份再分析

## 本工作流来源
2026-02-13 终端卡死事故复盘。AI 在 L1 反复循环了 4 轮（清空hooks→恢复hooks→改Blocking→再清空），
直到用户给出精准提示（"问题是全局的，追溯你之前改了什么全局配置"）才定位到根因。
