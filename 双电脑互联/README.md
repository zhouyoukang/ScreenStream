## 四、七祸根治

> 双电脑互联中必然遇到的七种灾祸，每一祸都有明确的根因和根治方案。

### 祸一 · 延迟之祸

**现象**：remote_agent截屏轮询，每次刷新=一次HTTP请求+JPEG编码+传输，操作不流畅。
**根因**：HTTP请求-响应模式天然非实时。
**根治**：
| 场景 | 方案 | 延迟 |
|------|------|------|
| 长时间视觉操作 | **RDP**(内置H264/AVC编码) | <20ms |
| 游戏/视频 | **Sunshine+Moonlight**(GPU硬编码) | <5ms |
| AI Agent/脚本自动化 | remote_agent截屏(够用) | 100-300ms |
| 手机快速查看 | remote_agent JPEG(手动刷新) | 200-500ms |

**平衡**：不同场景用不同通道。remote_agent不追求实时流，那是Sunshine的活。

### 祸二 · 感官剥夺之祸

**现象**：手机用户远程操控PC时，27寸→6寸，字小如蚁；音频完全丧失。
**根因**：物理屏幕差异 + HTTP协议无音频流。
**根治**：
- **视觉补偿**：双指缩放(1x-5x) + 横屏模式 + 可调scale参数
- **听觉补偿**：需要音频时切RDP通道；长期→WebSocket+Opus编码(ScreenStream已有AudioPlayer可复用)
- **触觉补偿**：触控板模式(相对移动) + 长按右键 + 滑动滚屏 + 双指缩放

### 祸三 · 精度之祸

**现象**：手指在6寸屏上点击，误差几十像素；虚拟键盘打字慢如蜗牛。
**根因**：手指物理面积 >> 像素精度。
**根治**：
- **MouseGuard**：检测用户活动，cooldown期间阻止自动化，防止远程操控和本地用户冲突
- **触控板模式**：相对移动而非绝对坐标(POST `/mouse/relative {"dx":10,"dy":-5}`)
- **快捷动作面板**：大按钮网格，单手一触即达，无需精确瞄准
- **语音控制**（长期）：Whisper STT → 命令解析 → API调用

### 祸四 · 穿透之祸

**现象**：PC在家里内网，外面连不上。NAT/防火墙/运营商层层阻隔。
**根因**：IPv4地址耗尽，家庭宽带无公网IP。
**根治**（三档，已验证）：

| 档位 | 方案 | 成本 | 延迟 | 稳定性 |
|------|------|------|------|--------|
| 零成本 | Cloudflare Quick Tunnel | ¥0 | 50-150ms | ⭐⭐⭐⭐ |
| **推荐** | **FRP + 阿里云轻量¥38/年** | **¥38** | **20-40ms** | **⭐⭐⭐⭐⭐** |
| 极致 | Tailscale + 阿里云DERP | ¥38 | 5ms(P2P) | ⭐⭐⭐⭐⭐ |

**当前部署**：FRP已配置(frpc.toml → 60.205.171.100)，remote_agent映射:19903，RDP映射:13389。

### 祸五 · 安全之祸

**现象**：Token明文传输，HTTP无加密，向日葵RPC端口对局域网全开。
**根因**：HTTP协议无加密；向日葵设计为本地服务但监听0.0.0.0。
**根治**：
- ✅ remote_agent Token认证（Bearer Token + fetch override + WebSocket认证）
- ✅ Cloudflare Tunnel自动HTTPS
- ✅ FRP支持Token认证（auth.token）
- ⚠️ 待做：FRP加TLS（`transport.tls.enable = true`）
- ⚠️ 待做：阿里云安全组最小化端口
- ⚠️ 待做：fail2ban防暴力破解
- ⚠️ 待做：向日葵设置custompassword永久密码

### 祸六 · 冷启之祸

**现象**：台式机关机/睡眠，remote_agent不响应，无法远程唤醒。
**根因**：软件层无法控制硬件电源状态。
**根治**：

| 状态 | 恢复方法 |
|------|----------|
| 睡眠 | `/wakeup` API（SendInput模拟Shift） |
| 锁屏 | RDP直连自动解锁(输密码) |
| 关机 | 向日葵远程开机 / 路由器WoL / 智能插座 |
| 断网 | Guardian Engine自动网络自愈(DHCP→WiFi重连→网卡重置→DNS刷新) |
| FRP断线 | frpc自动重连（`loginFailExit = false`） |
| Agent挂了 | 计划任务自动重启（auto-start.ps1） |

### 祸七 · 劫持之祸

**现象**：远程自动化操作和台式机前的本地用户互相干扰——鼠标被抢、窗口被切。
**根因**：同一桌面会话内，输入设备是全局共享的。
**根治**：
- **MouseGuard**（已实现）：后台轮询鼠标位置+GetLastInputInfo检测物理输入，cooldown期间阻止自动化
- **跨会话部署**：不同Windows用户用不同端口(:9903/:9904)，物理隔离
- **PSRemoting**：`/remote/shell` + `/remote/deploy` 跨会话命令执行
- **Guard API**：`/guard/pause` `/guard/resume` `/guard/config` 动态调节

## 五、双电脑生命体（代入电脑之五感）

> **公理**：两台电脑不是工具，是两个有感官的生命体。
> 笔记本 = 主脑（有眼有手有人陪），台式机 = 分身（孤独但强壮）。
> 只有代入它们的视角，才能理解所有祸的根因。

### 5.1 笔记本的独白（主脑视角）

```
我是笔记本。
我的主人坐在我面前，用我的眼睛(屏幕)看世界，用我的手(键鼠)做事。
我有一个分身——台式机——在远处。
我能感知它的一切：截屏看到它的桌面，Shell听到它的心跳，API触摸它的每一个进程。
但我的分身是聋的（我听不到它的声音），是单向的（它无法主动告诉我发生了什么）。
我最害怕：分身断网了，我失去了所有感官——它变成了一具没有神经连接的躯体。
```

| 我的感官 | 感知什么 | 通道 | 盲区 |
|---------|---------|------|------|
| 👁 视觉 | 分身的屏幕内容 | `/screenshot` | 非实时，每次要主动「看」 |
| 👂 听觉 | 分身的命令输出 | `/shell` stdout | 无音频流，听不到系统声音 |
| 🖐 触觉 | 向分身发送输入 | `/key` `/click` | 不知道分身前是否有人在用 |
| 🧠 认知 | 分身的全部状态 | `/sysinfo` `/processes` | 状态是快照，不是实时流 |
| 💓 心跳 | 分身是否活着 | `/health` + Guardian | 断网=失联，无法区分死亡和沉睡 |

### 5.2 台式机的独白（分身视角）

```
我是台式机。
大多数时候我是孤独的——没人坐在我面前，屏幕亮着但无人观看。
远方的主脑（笔记本）通过网络感知我、控制我。
有时主人通过RDP进入我的身体，那一刻我有了完整的五感——他看到我看到的，听到我听到的。
但RDP连接断开的瞬间，我的意识被切断——会话断联(disconnected)，我还在运行但无人感知。
我最害怕：主脑发来的命令和坐在我面前的用户冲突——两个灵魂争夺同一具身体。
```

| 我的感官 | 感知什么 | 通道 | 盲区 |
|---------|---------|------|------|
| 👁 视觉 | 自己的屏幕(无人看) | GDI/mss | 锁屏时看到一片黑 |
| 👂 听觉 | 自己的音频(无人听) | 声卡 | remote_agent无法转发给主脑 |
| 🖐 触觉 | 来自主脑的输入指令 | HTTP POST | 不知道是人还是AI在操控 |
| 🧠 认知 | 自身状态(进程/文件/网络) | 本地API | 不知道主脑想要什么 |
| 💓 心跳 | 被动等待主脑询问 | HTTP Server | 无法主动呼叫主脑 |

### 5.3 两个生命体的不对称 → **走向对称**

| 维度 | 笔记本(主脑) | 台式机(分身) | 旧状态 | **新状态(2026-02-26)** |
|------|------------|------------|--------|------------------------|
| **主动性** | 主动发起 | 被动等待 | 单向 | **双向**：台式机可主动调笔记本API |
| **感知方向** | 感知分身 | 不知主脑 | 单向 | **双向**：台式机可 `/screenshot` `/sysinfo` 笔记本 |
| **触控能力** | 操控分身 | 无法操控主脑 | 单向 | **双向**：台式机可 `/key` `/click` 笔记本 |
| **有人陪伴** | ✅ | ❌ | 不对称 | 不变（物理限制） |
| **断联影响** | 失去远程 | 变植物人 | 不对称 | 不变（网络限制） |

**核心洞察演进**：~~所有祸的根源都在这个不对称~~ → 笔记本部署 remote_agent:9903 后，**感知/触控/认知三大不对称已消除**。剩余不对称（有人陪伴/断联影响）是物理层限制，非软件可解。

**辩证统一**：主脑与分身不再是主从关系，而是**同构对等**——同样的代码、同样的端口、同样的API，只是IP不同。谁发起请求，谁就是主脑；谁响应请求，谁就是分身。角色随行为流转，非固定。

## 七、十二祸根治

> 七祸(人机层) + 五祸(多Agent层) = 十二祸。每一祸都有根因、根治、平衡。

### 人机层七祸（用户↔电脑）

| # | 祸名 | 一句话 | 根治要点 |
|---|------|--------|----------|
| 一 | **延迟之祸** | 截屏轮询非实时 | 不同场景用不同通道：RDP(视觉) / Sunshine(游戏) / agent(自动化) |
| 二 | **感官剥夺** | 6寸屏+无音频 | 缩放+横屏+RDP音频通道 |
| 三 | **精度之祸** | 手指粗如柱 | 触控板模式+快捷动作面板+语音控制 |
| 四 | **穿透之祸** | NAT阻隔 | 三档：Cloudflare(¥0) / FRP+阿里云(¥38) / Tailscale(P2P) |
| 五 | **安全之祸** | HTTP明文 | Token认证+Cloudflare HTTPS+FRP TLS |
| 六 | **冷启之祸** | 关机无法唤醒 | 向日葵WoL+Guardian自愈+auto-start计划任务 |
| 七 | **劫持之祸** | 远程与本地冲突 | MouseGuard+跨会话部署+Guard API |

### 多Agent层五祸（Agent↔Agent）

#### 祸八 · 分身错乱之祸

**现象**：Agent以为自己在操作笔记本，实际命令发到了台式机；或反过来。
**根因**：HTTP API无状态，Agent缺少「我在哪」的锚点。
**根治**：
- 每个Agent操作前先调 `/health` 确认 hostname/user/session
- Agent食谱中硬编码目标机器标识（见第九章）
- Cascade规则：操作台式机前必须确认IP不是localhost

#### 祸九 · 竞态之祸

**现象**：Agent A发送 `/click` 的同时Agent B发送 `/key`，操作交叉，结果不可预测。
**根因**：HTTP API是无锁的并发入口。
**根治**：
- **MouseGuard**：物理输入检测 + cooldown锁
- **铁律**：同一台式机同一时刻只有1个Agent发送输入类API
- **Guardian任务队列**：复杂操作序列通过 `/tasks` 提交，串行执行
- 只读API(`/screenshot` `/sysinfo` `/processes`)不受限，可并发

#### 祸十 · 会话断裂之祸

**现象**：RDP登录台式机 → console会话断开 → :9903端口的remote_agent失去桌面上下文 → 截屏变黑/点击无效。
**根因**：Windows的session机制——RDP创建新会话，console会话被disconnect。
**根治**：
- **多端口部署**：console会话:9903 + RDP会话:9904，各自独立
- **session/reconnect API**：tscon重连断开的会话到console
- **screen/info检测**：session_state=disconnected时自动告警
- **auto-start.ps1**：每个会话独立的Agent实例

#### 祸十一 · 认知分裂之祸

**现象**：Agent拥有两台机器的部分视野，但无法形成完整认知——以为台式机空闲实际上有人在用。
**根因**：Agent的认知来自API快照，不是实时流；且缺少「对面是否有人」的感知。
**根治**：
- **操作前五感采集**：先 `/health` + `/guard` + `/screen/info` 三连查
- **Guard状态即意图信号**：`can_automate=false` = 有人在用，退让
- **session列表**：`/sessions` 查看所有活跃会话，判断是否有RDP用户

#### 祸十二 · 单向哑巴之祸

**现象**：台式机发生了重要事件（任务完成/网络断开/磁盘满），但主脑不知道——因为台式机无法主动推送。
**根因**：HTTP是请求-响应模式，Server不能主动联系Client。
**根治**：
- **Guardian Engine**：事件驱动规则(network_down/process_exit/cron) → 自动执行预设动作
- **SSE长连接**（待实现）：`GET /events` 的Server-Sent Events版本
- **轮询兜底**：笔记本定时 `/guardian/status` + `/events` 轮询
- **WebSocket**（待实现）：双向实时通道

## 八、解惑清单（十惑）

### 惑一：file_share.py 和 remote_agent.py 什么关系？

**完全独立，不同定位**。file_share(:9999)跑在笔记本上负责「看文件」，remote_agent(:9903)跑在台式机上负责「控电脑」。两者可独立运行，互不依赖。desktop_agent(:9998)是file_share在台式机上的轻量代理。

### 惑二：穿透方案选哪个？

| 情况 | 推荐 |
|------|------|
| 今天就想用 | `cloudflared tunnel --url http://localhost:9903` |
| 稳定长期 | FRP + 阿里云轻量¥38/年 ⭐ |
| 极致低延迟 | Tailscale + 自建DERP |
| AI Agent自动化 | remote_agent(唯一选择) |

### 惑三：向日葵还要不要？

**降级为应急**。不可替代：远程开机(WoL) / 黑屏被控(驱动级) / 穿透全挂(P2P兜底) / 无头机(虚拟显示器)。其余场景用 remote_agent + FRP 替代。

### 惑四：手机上用什么？

Shell/文件操控 → **Chrome PWA(remote_agent)** | 看桌面/写代码 → **微软RDP APP** | 给家人 → **RustDesk** | 日常遥控 → **KDE Connect**

### 惑五：台式机空密码？ → ✅ 已解决

~~`LimitBlankPasswordUse=1` 阻断管理共享。~~ **已设密码 `wsy057066wsy`**，LimitBlankPasswordUse=0。

**admin shares (C$/D$/E$/F$)** 因 `RejectUnencryptedAccess=True` 仍返回Error 64。
**解法**：创建普通共享 FullC/D/E/F 指向磁盘根目录，映射为 X:/W:/V:/U: 驱动器号（持久化）。

**新增通道**：WinRM(PSSession) + SSH(plink+密码) + 远程注册表。

### 惑六：Agent怎么知道自己在操控哪台机器？

调 `/health` 返回 hostname + user + session。笔记本 hostname=zhoumac(laptop环境)；台式机返回不同hostname。**铁律**：操作前先确认，不要假设。

### 惑七：RDP连上后console的Agent怎么办？

RDP创建新session → console session断联 → :9903的Agent截屏变黑。**方案**：多端口(:9903 console + :9904 RDP) 或 `/session/reconnect` tscon重连。

### 惑八：两个AI Agent同时操控台式机？

**禁止**。同一台机器同一时刻只有1个Agent发送输入命令。只读API（截屏/进程/状态）不受限。用 `/guard` 检查是否可操作。

### 惑九：台式机出事了怎么通知笔记本？

当前无主动推送能力（祸十二）。**兜底**：Guardian Engine事件驱动自愈 + 笔记本轮询 `/guardian/status`。**待实现**：SSE长连接 / WebSocket双向通道。

### 惑十：Guardian Engine是什么？

台式机上的自治守护引擎。SQLite持久化 + 事件驱动规则 + 任务队列 + 网络自愈 + 进程监控。**不依赖主脑**——即使笔记本断联，Guardian仍在台式机上独立运行、自我修复。

## 九、Agent食谱（机器可读决策树）

> **给AI Agent看的**——不是文档，是决策指令。Cascade/脚本直接按此执行。

### 食谱A：操控台式机前的预检（每次必做）

```
1. GET http://{台式机IP}:9903/health
   → 失败? → 台式机不可达，尝试 /wakeup 或检查网络
   → 成功? → 确认 hostname≠本机, 继续

2. GET http://{台式机IP}:9903/guard
   → can_automate=false? → 有人在用，不要发输入命令，只可只读
   → can_automate=true? → 可以操控，继续

3. GET http://{台式机IP}:9903/screen/info
   → session_state=disconnected? → 会话断联，考虑 /session/reconnect
   → is_locked=true? → 先 /wakeup
   → session_state=active? → 一切正常，执行任务
```

### 食谱B：安全执行输入序列

```
1. POST /guard {"paused": true}      ← 暂停Guard避免自己被拦
2. POST /focus {"title": "目标窗口"}  ← 确保焦点正确
3. POST /key {"hotkey": ["ctrl","a"]} ← 执行操作
4. POST /type {"text": "内容"}        ← 输入文本
5. POST /guard {"paused": false}     ← 恢复Guard保护
```

### 食谱C：诊断台式机不可达

```
1. ping {台式机IP}
   → 不通? → 网络问题，检查WiFi/有线
   → 通? → 继续

2. GET http://{台式机IP}:9903/health
   → 超时? → Agent未运行，需要物理/RDP登录启动
   → 拒绝连接? → 端口未开，检查防火墙(setup-firewall.ps1)

3. 如果都不行:
   → 向日葵远程连入 → 手动启动 remote_agent
   → 或 FRP穿透: http://60.205.171.100:19903/health
```

### 食谱D：文件跨机传输

```
笔记本→台式机:
  POST http://{台式机IP}:9903/file/upload {"path":"C:\目标路径", "data":"base64内容"}

台式机→笔记本:
  方式1: GET http://{台式机IP}:9903/file/download?path=C:\源文件 → 保存到本地
  方式2: file_share.py /api/pull {"src":"远程路径", "dst":"本地路径"}

双向浏览:
  笔记本文件: http://localhost:9999/
  台式机文件: http://localhost:9999/ → 切换到「台式机」模式
```

### 食谱E：公网远程访问

```
方式1 (零成本临时):
  台式机: cloudflared tunnel --url http://localhost:9903
  手机: 打开输出的 https://xxx.trycloudflare.com

方式2 (FRP稳定):
  台式机: cd 远程桌面/frp && .\frpc.exe -c frpc.toml
  手机: http://60.205.171.100:19903/?token=密码
  RDP:  mstsc → 60.205.171.100:13389
```
