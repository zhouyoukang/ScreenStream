## 〇、五感感知现场

### 用户症状（第一人称还原）
- **视**：正在远程操控电脑时，剪切板内容不断自动变化

- **听**：Samsung手机和平板同时收到通知/剪切板更新

## 二、☰乾·天道（谁是剪切板事件的源头发起者？）

> 乾为天、为首、为创造。追溯：谁第一个改变了剪切板？

### 源头分析：远控场景下的触发链

当用户通过ToDesk远程控制本机时：

```
远程主控端复制文本
  ↓ ToDesk协议
本机ToDesk进程收到 → 调用 Win32 SetClipboardData()
  ↓ Windows广播 WM_CLIPBOARDUPDATE
所有监听者同时收到通知（见下方☳震）
```

**关键发现**: 源头不是一个，而是**多源头并发**：
1. **ToDesk**: 远程主控端的每次复制都会同步到被控端
2. **向日葵**: 如果同时有向日葵连接，也会独立写剪切板
3. **Phone Link**: 手机端的复制操作也会通过CDP同步到PC
4. **Samsung DeX**: Note20 Ultra的复制操作也会同步到PC

> 🔴 **乾卦之祸**: 四个独立源头各自为政，没有协调机制

## 三、☷坤·地载（谁在被动接收剪切板变化？）

> 坤为地、为承、为被动。谁在无差别地监听剪切板？

### Win32 剪切板监听机制

Windows提供两种API让程序监听剪切板变化：
1. `AddClipboardFormatListener()` → 接收 `WM_CLIPBOARDUPDATE` 消息
2. `SetClipboardViewer()` → 旧API，链式通知

**所有10个实体都注册了监听**。当任何一个实体写入剪切板：

```
SetClipboardData() 被调用
  ↓
Windows内核广播 WM_CLIPBOARDUPDATE
  ↓ （同时送达所有监听者）
  ├→ ToDesk: 检测到变化 → 发送到远程主控端
  ├→ 向日葵: 检测到变化 → 发送到向日葵主控端
  ├→ Samsung DeX: 检测到变化 → 发送到Note20 Ultra
  ├→ PhoneExperienceHost: 检测到变化 → 通过CDP发送到所有关联设备
  ├→ CrossDeviceService: 检测到变化 → 通过微软云同步
  ├→ cbdhsvc: 检测到变化 → 记录剪切板历史 + 可能云同步
  ├→ Samsung Notes: 检测到变化 → 可能记录到便签
  └→ nblink: 检测到变化 → 可能通过文件同步传播
```

> 🔴 **坤卦之祸**: 所有监听者**无差别接收**，不区分"用户主动复制"还是"程序回写"

## 四、☳震·雷动（什么触发了剪切板死循环？）

> 震为雷、为动、为激发。循环的触发点在哪里？

### 死循环核心机制：监听-传播-回写

```
A写剪切板 → Windows广播 → B收到 → B传播到外部设备
                                  → 外部设备回传 → B写剪切板
                                                    ↓
                              Windows广播 → A收到 → A传播到外部设备
                                                    → 外部设备回传 → A写剪切板
                                                                      ↓
                                                    Windows广播 → (循环)
```

### 具体死循环路径（已识别 5 条）

| 循环# | 路径 | 严重度 |
|--------|------|--------|
| **L1** | ToDesk写入 → DeX监听 → DeX发到手机 → 手机回传 → DeX写入 → ToDesk监听 → ... | 🔴致命 |
| **L2** | ToDesk写入 → PhoneLink监听 → PhoneLink发到手机 → 手机回传 → PhoneLink写入 → ToDesk监听 → ... | 🔴致命 |
| **L3** | 向日葵写入 → ToDesk监听 → ToDesk发到远端 → 远端回传 → ToDesk写入 → 向日葵监听 → ... | 🔴致命 |
| **L4** | DeX写入 → PhoneLink监听 → PhoneLink发到另一设备 → 设备回传 → PhoneLink写入 → DeX监听 → ... | 🟡高 |
| **L5** | cbdhsvc云同步 → 本地写入 → CrossDevice监听 → CDP传播 → 另一设备回传 → 写入 → ... | 🟡高 |

> 🔴 **震卦之祸**: 5条独立死循环同时运行，**指数级放大**剪切板事件

## 五、☴巽·风渗（剪切板内容如何传播到手机/平板？）

> 巽为风、为入、为渗透。内容如何从PC蔓延到所有设备？

### 传播通道矩阵（PC → 移动设备）

| 通道 | 协议 | 目标设备 | 延迟 | 可否关闭剪切板 |
|------|------|---------|------|--------------|
| **Samsung DeX** | WiFi Direct/USB | Note20 Ultra | ~50ms | ❌ 无独立开关 |
| **ss_conn_service** | Loopback→WiFi | Samsung全设备 | ~100ms | ❌ 只能停服务 |
| **Phone Link (CDP)** | Microsoft Cloud | 关联的Samsung设备 | ~200ms | ✅ 有设置项 |
| **CrossDeviceService** | Microsoft Cloud | 所有Windows设备 | ~300ms | ✅ 有设置项 |
| **cbdhsvc云同步** | Microsoft Cloud | 同MS账号设备 | ~500ms | ✅ 有设置项 |

### 传播到Samsung手机/平板的路径

```
PC剪切板变化
  ├─[通道1] Samsung DeX → Note20 Ultra (直连, 最快)
  ├─[通道2] PhoneLink → Note20 Ultra (CDP, 重复!)
  ├─[通道3] CrossDevice → Note20 Ultra (CDP, 三重!)
  └─[通道4] cbdhsvc → Microsoft Cloud → 同账号设备

Note20 Ultra 收到3份相同的剪切板更新
  ├→ 显示3次通知
  ├→ Samsung生态同步到平板(Samsung Cloud/Flow)
  └→ 平板也收到通知
```

> 🔴 **巽卦之祸**: **同一份内容通过3-4条独立通道重复传播**到同一设备

## 六、☵坎·水险（隐藏的陷阱和风险）

> 坎为水、为险、为陷。看不见的深层风险。

### 陷阱1: Samsung DeX的"幽灵自启"
```
注册表显示: AutoConnect=0, Autorun=0 (已关闭)
但实际: 进程正在运行! PID 7016, 启动于15:34
原因: CommandArguments="--autorun" → 被其他机制(服务/计划任务/ss_conn_service)拉起
```
**注册表改了但进程没死** — 这是上次修复未彻底的残留。

### 陷阱2: 两个远控软件的"双重写入"
```
ToDesk (3进程, 974 CPU秒 — 说明极其活跃)
向日葵 (3+1进程, autorun=1, enableremotecontrol=1)
```
两个远控软件**同时运行**。即使只有ToDesk在被远控，向日葵也在监听剪切板。

### 陷阱3: Phone Link的609MB内存巨兽
```
PhoneExperienceHost: 609MB内存, 连接20.195.65.193
```
这个进程是所有参与者中**内存最大**的，说明它缓存了大量跨设备数据，包括剪切板历史。

### 陷阱4: 节点小宝(nblink)的隐蔽参与
```
6个进程, 196MB总内存, 连接阿里云多台服务器
NodeBabyLinkBackup — 备份服务(可能同步剪切板内容)
NodeBabyLinkRfile — 文件服务(可能传输剪切板文件)
NodeBabyLinkDevice — 设备服务(管理远程设备)
```
**新发现**: 这是上次分析中未识别的第7个参与者。

### 陷阱5: Windows CDP的"默认开启"
```
RomeSdkChannelUserAuthzPolicy = 1 (允许跨设备)
CdpSessionUserAuthzPolicy = 1 (允许跨设备会话)
CloudClipboard: 未显式设置 → 可能默认开启
```
Windows的跨设备同步是**默认开启**的，用户可能从未意识到。

## 七、☶艮·山止（哪里可以阻断？阻断点分析）

> 艮为山、为止、为界。找到最少的阻断点实现最大效果。

### 阻断层次分析

```
Layer 0: Windows内核 (WM_CLIPBOARDUPDATE广播)
  → 不可阻断，这是OS基础设施

Layer 1: 监听者注册 (AddClipboardFormatListener)
  → 每个程序自行注册，需逐个关闭

Layer 2: 监听者的"写回"行为
  → 这是死循环的关键环节
  → 如果监听者只读不写，循环就断了

Layer 3: 外部传播通道
  → 关闭网络同步通道
  → 保留本地功能但切断跨设备传播

Layer 4: 设备端接收
  → Samsung手机/平板端关闭接收
```

### 最小阻断集（理论最优解）

只需在 **Layer 2** 切断4个写回路径：
1. 停止Samsung DeX的剪切板写回
2. 停止Phone Link的剪切板写回
3. 停止向日葵的剪切板监听
4. 停止Windows云剪切板同步

保留ToDesk的剪切板同步（因为这是用户实际需要的远控功能）。

### 根因公式

```
剪切板风暴 = Σ(源头数) × Σ(监听者数) × Σ(传播通道数) × 1/(阻断机制数)
           = 4 × 10 × 4 × 1/0
           = ∞ (无穷循环)
```

**阻断机制数 = 0** 是核心问题。没有任何一个程序在写剪切板前检查"这个变化是不是我自己刚才同步过来的"。

## 总结（暂不修复，仅记录发现）

**核心矛盾**: 10个实体×5条死循环×4条重复传播通道 vs 用户只需要1条同步通道。

**哲学本质**: 这不是任何单一程序的bug，而是**涌现问题**(emergent problem) — 每个程序各自正确运行，但组合在一起产生了灾难性的系统行为。正如老子所言："天下皆知美之为美，斯恶已" — 每个程序都想"为用户同步剪切板"这个"美"事，结果造成了"恶"(系统性风暴)。
